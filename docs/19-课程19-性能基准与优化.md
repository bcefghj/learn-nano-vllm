## 一、概念讲解：LLM 推理性能指标

### 1.1 为什么性能测试很重要

LLM 推理的性能直接影响：

- **用户体验**：首字延迟（Time To First Token, TTFT）决定用户等待时间。
- **服务成本**：吞吐量越高，单位 token 的 GPU 成本越低。
- **可扩展性**：能否在有限 GPU 资源下服务更多用户。

因此，**科学、可复现的性能测试** 是评估推理引擎质量的基石。

### 1.2 核心性能指标

| 指标 | 英文 | 定义 | 典型单位 |
|------|------|------|---------|
| 首字延迟 | TTFT (Time To First Token) | 从发送请求到收到第一个生成 token 的时间 | ms |
| 字间延迟 | TPOT (Time Per Output Token) | decode 阶段每生成一个 token 的平均时间 | ms |
| 端到端延迟 | E2E Latency | 从发送请求到收到完整响应的时间 | ms / s |
| Prefill 吞吐量 | Prefill Throughput | 每秒处理的 prompt token 数 | tokens/s |
| Decode 吞吐量 | Decode Throughput | 每秒生成的新 token 数 | tokens/s |
| 总吞吐量 | Total Throughput | 每秒处理的总 token 数（prefill + decode） | tokens/s |

### 1.3 Prefill 吞吐量 vs Decode 吞吐量

这是面试中最常考的区分：

**Prefill 吞吐量**：
- 衡量引擎处理 prompt 的速度。
- 与 **计算能力（FLOPS）** 相关：prefill 是计算密集型。
- 长 prompt 时更能体现差异。

**Decode 吞吐量**：
- 衡量引擎生成新 token 的速度。
- 与 **内存带宽** 和 **调度效率** 相关：decode 是内存密集型。
- 受 batch size 影响大：batch 越大，GPU 利用率越高，decode 吞吐量越高。

两者关系：

\[
\text{Total Throughput} = \frac{\text{prefill\_tokens} + \text{decode\_tokens}}{\text{total\_time}}
\]

## 二、源码对照：bench.py 完整解读

仓库根目录 `bench.py`（示例使用本机路径 `~/huggingface/Qwen3-0.6B/`、`enforce_eager=False`）与下列结构一致。该脚本测的是 **decode 侧按 `max_tokens` 约定总生成量** 下的平均吞吐（见 `total_tokens` 计算），而非单独拆分 prefill/decode 的 micro-benchmark。

### 2.1 完整代码分析

```python
def main():
    # ① 设置随机种子，保证可复现
    seed(0)

    # ② 生成测试数据
    num_seqs = 256
    prompt_token_ids = [
        [randint(0, 10000) for _ in range(randint(100, max_input_len))]
        for _ in range(num_seqs)
    ]
    sampling_params = [
        SamplingParams(temperature=0.6, ignore_eos=True, max_tokens=randint(100, max_ouput_len))
        for _ in range(num_seqs)
    ]

    # ③ Warmup：预热引擎
    llm.generate(["Benchmark: "], SamplingParams())

    # ④ 正式测试
    t = time.time()
    llm.generate(prompt_token_ids, sampling_params, use_tqdm=False)
    throughput = total_tokens / (time.time() - t)
```

### 2.2 逐段解析

#### 随机种子

```python
seed(0)
```

设置随机种子的目的：

- **可复现性**：不同运行之间使用相同的 prompt 长度和输出长度分布。
- **公平对比**：对比不同框架或不同优化配置时，输入数据完全一致。
- 影响的随机源：prompt 中的 token ID、prompt 长度、输出长度。

#### 测试数据生成

```python
num_seqs = 256
prompt_token_ids = [
    [randint(0, 10000) for _ in range(randint(100, max_input_len))]
    for _ in range(num_seqs)
]
```

| 参数 | 说明 |
|------|------|
| `num_seqs = 256` | 测试 256 条请求，模拟并发场景 |
| `randint(100, max_input_len)` | prompt 长度在 100 到 max_input_len 之间随机 |
| `randint(0, 10000)` | token ID 随机，覆盖词表的一部分 |

为什么用 **随机 token ID** 而非真实文本？

- **消除 tokenize 开销**：直接传入 token ID 列表，跳过 tokenizer。
- **控制变量**：不同 prompt 文本的 tokenize 结果长度不一，用随机 ID 可以精确控制。
- **实际性能无关文本内容**：推理引擎的计算速度不依赖于 token 的具体值，只依赖于形状。

#### 采样参数

```python
sampling_params = [
    SamplingParams(temperature=0.6, ignore_eos=True, max_tokens=randint(100, max_ouput_len))
    for _ in range(num_seqs)
]
```

关键设置：

- `temperature=0.6`：使用概率采样（非 greedy），更接近生产环境。
- `ignore_eos=True`：**忽略 EOS token**，强制生成到 `max_tokens`。这是 benchmark 的标准做法，避免不同运行之间因为 EOS 位置不同导致生成长度不一致。
- `max_tokens=randint(100, max_ouput_len)`：输出长度随机，模拟真实场景下不同请求的输出长度差异。

#### Warmup（预热）

```python
llm.generate(["Benchmark: "], SamplingParams())
```

为什么需要 warmup？

1. **CUDA 上下文初始化**：首次使用 GPU 时需要初始化驱动、分配设备内存。
2. **torch.compile JIT 编译**：`@torch.compile` 修饰的函数在首次调用时编译，后续调用使用缓存。
3. **CUDA Graph capture**：已在 ModelRunner 初始化时完成，但首次 replay 可能有额外开销。
4. **cuBLAS handle 初始化**：首次矩阵乘会创建和配置 cuBLAS 句柄。
5. **内存分配器预热**：PyTorch 的 CUDA 内存分配器在首次分配后会缓存内存块。

如果不 warmup，首次测试的数据会包含这些一次性开销，导致 **低估真实性能**。

#### 正式测试与吞吐量计算

```python
t = time.time()
llm.generate(prompt_token_ids, sampling_params, use_tqdm=False)
t = (time.time() - t)
total_tokens = sum(sp.max_tokens for sp in sampling_params)
throughput = total_tokens / t
print(f"Total: {total_tokens}tok, Time: {t:.2f}s, Throughput: {throughput:.2f}tok/s")
```

- `use_tqdm=False`：关闭进度条，避免 tqdm 的 IO 操作影响计时。
- **`total_tokens = sum(sp.max_tokens for sp in sampling_params)`**：在 `ignore_eos=True` 下，各序列生成长度被钳制为 `max_tokens`，故总 **生成 token 工作量** 可事先求和；该指标与 **prefill 输入 token 总数** 无关，因此本脚本打印的 **tok/s** 偏向 **decode 总产出/总时间** 的宏观指标（仍含各请求 prefill 时间，故是端到端混合吞吐）。
- `time.time()` 测的是 **wall-clock time**（挂钟时间），包含了所有开销。

### 2.3 测试方法论

一个严谨的 benchmark 应该考虑：

| 要素 | bench.py 的做法 | 理想做法 |
|------|----------------|---------|
| 可复现性 | `seed(0)` 固定随机种子 | ✅ |
| Warmup | 运行一次短请求 | ✅ |
| 数据规模 | 256 条请求 | 足够 |
| 输入多样性 | 随机长度 prompt | ✅ |
| 输出多样性 | 随机长度输出 | ✅ |
| 关闭 EOS | `ignore_eos=True` | ✅ |
| 多次重复 | 未做 | 理想情况应跑 3-5 次取平均 |
| GPU 独占 | 未显式检查 | 应确保无其他进程占用 GPU |
| 温度测量 | 未监测 | 生产环境需监测 GPU 温度（热降频影响性能） |

## 三、各项优化技术的加速效果

### 3.1 CUDA Graph 加速（回顾第16课）

**原理**：将 decode 阶段的多次 kernel launch 合并为一次 graph launch，降低 CPU 开销。

**加速效果**（以 Qwen2.5-7B 在 A100 上为例）：

| 场景 | 无 Graph | 有 Graph | 加速比 |
|------|---------|---------|--------|
| Decode 吞吐量 (bs=1) | ~800 tok/s | ~1500 tok/s | ~1.9x |
| Decode 吞吐量 (bs=32) | ~2500 tok/s | ~3800 tok/s | ~1.5x |
| Decode 吞吐量 (bs=128) | ~4000 tok/s | ~5500 tok/s | ~1.4x |

**规律**：batch 越小，CUDA Graph 加速比越大。这是因为小 batch 时 kernel launch overhead 占总耗时的比例更高。

**何时不应使用**：
- 调试阶段（`enforce_eager=True`）。
- batch 超过 512（超出 capture 上界）。
- 需要逐 kernel profiling 时。

### 3.2 `torch.compile` 加速

**原理**：PyTorch 2.0 引入的编译器，将 Python + PyTorch 代码编译为优化的 GPU kernel。

**主要优化**：
1. **算子融合**：将多个小算子合并为一个大 kernel，减少 launch 和内存访问。
2. **内存优化**：减少中间张量的分配。
3. **自动向量化**：利用 GPU 的向量指令。
4. **后端代码生成**：通过 Inductor 后端生成 Triton kernel。

**nano-vllm 中使用 `torch.compile` 的典型位置**（以 `nanovllm/layers/sampler.py` 为例）：

```python
class Sampler(nn.Module):
    @torch.compile
    def forward(self, logits, temperatures):
        logits = logits.float().div_(temperatures.unsqueeze(dim=1))
        probs = torch.softmax(logits, dim=-1)
        sample_tokens = probs.div_(torch.empty_like(probs).exponential_(1).clamp_min_(1e-10)).argmax(dim=-1)
        return sample_tokens
```

| 位置 | 功能 | 优化效果 |
|------|------|---------|
| `Sampler.forward` | 温度缩放 + softmax + Gumbel-max 采样 | 2-3x 加速（相对未编译） |
| 模型中其它带 `@torch.compile` 的小算子（若实现中有） | RMSNorm、RoPE、SiLU 等 | 融合后减少 launch |

**加速效果（整体）**：

| 指标 | 无 compile | 有 compile | 加速比 |
|------|-----------|-----------|--------|
| 总吞吐量 | ~3000 tok/s | ~3800 tok/s | ~1.27x |

**注意**：`torch.compile` 的首次编译可能需要数十秒，但编译结果会被缓存。

### 3.3 FlashAttention 加速

**原理**：通过 **分块计算（Tiling）** 和 **在线 Softmax（Online Softmax）** 减少 HBM（高带宽内存）访问，将注意力计算的 IO 复杂度从 \(O(N^2)\) 降低到 \(O(N)\)。

**核心技术**：

```
传统 Attention：
  S = Q @ K^T       →  写入 HBM（N×N 矩阵）
  P = softmax(S)    →  从 HBM 读取 S，写入 P
  O = P @ V         →  从 HBM 读取 P

FlashAttention：
  将 Q、K、V 分块 → 在 SRAM 中完成 S、P 计算 → 直接累加到 O
  不需要在 HBM 中存储完整的 N×N 注意力矩阵
```

**加速效果（Prefill 阶段最显著）**：

| Prompt 长度 | 标准 Attention | FlashAttention | 加速比 |
|------------|---------------|----------------|--------|
| 512 | 5 ms | 2 ms | 2.5x |
| 2048 | 80 ms | 15 ms | 5.3x |
| 8192 | 1280 ms | 120 ms | 10.7x |

**规律**：序列越长，FlashAttention 的优势越大，因为避免了 \(O(N^2)\) 的中间矩阵存储。

**nano-vllm 中的使用**：
- Prefill：使用 `flash_attn_varlen_func`，支持变长序列批处理。
- Decode：使用 `flash_attn_with_kvcache`，直接从 KV Cache 读取历史 K/V。

### 3.4 前缀缓存（Prefix Caching）加速

**原理**：当多个请求共享相同的前缀（如系统 prompt）时，**复用** 已计算的 KV Cache，避免重复计算。

**典型场景**：

```
请求 1：[系统提示词] + "请问天气如何？"
请求 2：[系统提示词] + "请帮我翻译一段话。"
请求 3：[系统提示词] + "写一首诗。"
```

三个请求共享相同的 [系统提示词]，开启前缀缓存后，系统提示词的 KV 只需计算一次。

**加速效果**：

| 场景 | 无前缀缓存 | 有前缀缓存 | 加速比 |
|------|-----------|-----------|--------|
| 共享 2048 token 前缀 | Prefill 35ms | Prefill 5ms | 7x |
| 共享 512 token 前缀 | Prefill 12ms | Prefill 5ms | 2.4x |
| 无共享前缀 | 无变化 | 无变化 | 1x |

**nano-vllm 的前缀缓存实现**（回顾之前课程）：
- 使用 **hash 表** 缓存已计算的 KV block。
- 按 block 粒度匹配前缀（`block_size` 个 token 为一组）。
- 命中缓存后跳过对应 token 的 prefill 计算。

## 四、nano-vllm vs vLLM 性能对比

### 4.1 对比维度

| 维度 | nano-vllm | vLLM |
|------|-----------|------|
| 代码量 | ~3000 行 | ~100K+ 行 |
| 支持模型 | Qwen2/Llama 等少数 | 数十种模型架构 |
| 量化 | 不支持 | AWQ/GPTQ/FP8 等 |
| Speculative Decoding | 不支持 | 支持 |
| Chunked Prefill | 不支持 | 支持 |
| 张量并行 | 基础支持 | 完整支持 |
| 流水线并行 | 不支持 | 支持 |

### 4.2 性能对比（以 Qwen2.5-7B 在单 A100 为例）

| 指标 | nano-vllm | vLLM | 比值 |
|------|-----------|------|------|
| Decode 吞吐量 | ~5000 tok/s | ~5500 tok/s | ~91% |
| Prefill 吞吐量 | ~8000 tok/s | ~9000 tok/s | ~89% |
| TTFT (bs=1) | ~50 ms | ~45 ms | ~111% |
| 内存占用 | 略高 | 优化更精细 | — |

nano-vllm 在核心推理性能上达到了 vLLM 的 **85-95%**，考虑到仅 3000 行代码，这是非常优秀的成果。

### 4.3 性能差距来源分析

nano-vllm 与 vLLM 的差距主要来自：

1. **Chunked Prefill 缺失**：vLLM 可以将长 prompt 分块处理，与 decode 交错执行，提高 GPU 利用率。
2. **调度策略简化**：nano-vllm 的调度器不如 vLLM 精细（如缺少 priority 调度、preemption 机制）。
3. **Paged Attention V2 缺失**：vLLM 使用了更优化的分页注意力实现。
4. **量化支持缺失**：AWQ/GPTQ 等量化可以减少内存带宽需求，提升 decode 速度。
5. **更精细的内存管理**：vLLM 的 BlockManager 有更细粒度的 swap/recompute 策略。

## 五、性能调优建议

### 5.1 硬件选择

| GPU | 显存 | 内存带宽 | FP16 算力 | 适合场景 |
|-----|------|---------|----------|---------|
| A100 80GB | 80GB | 2.0 TB/s | 312 TFLOPS | 大模型、长上下文 |
| A100 40GB | 40GB | 1.5 TB/s | 312 TFLOPS | 7B-13B 模型 |
| H100 | 80GB | 3.35 TB/s | 989 TFLOPS | 最高性能 |
| L40S | 48GB | 864 GB/s | 362 TFLOPS | 性价比推理 |
| A10 | 24GB | 600 GB/s | 125 TFLOPS | 小模型推理 |

### 5.2 Batch Size 调优

Batch size 是影响吞吐量最关键的参数之一：

```
吞吐量
  ↑
  │         ┌────────── 内存带宽瓶颈线
  │        ╱
  │      ╱
  │    ╱
  │  ╱
  │╱
  └──────────────→ Batch Size
  1   4   16  64  256
```

- **bs 较小**（1-8）：GPU 利用率低，内存带宽未饱和，吞吐量低。
- **bs 中等**（16-128）：逐渐接近内存带宽上限，吞吐量快速增长。
- **bs 较大**（128+）：接近显存上限，KV Cache 占用大量内存。

**建议**：设 `max_num_seqs` 为能容纳在 GPU 显存中的最大值。

### 5.3 关键参数调优清单

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `max_num_seqs` | 根据显存计算 | 越大吞吐量越高，但受显存限制 |
| `gpu_memory_utilization` | 0.85-0.95 | 留余量避免 OOM，但不宜过低 |
| `enforce_eager` | False | 除非调试，否则应启用 CUDA Graph |
| `enable_prefix_caching` | True（如适用） | 共享前缀场景下显著加速 |
| `block_size` | 16 | 通常无需修改 |

### 5.4 Profile 工具

性能分析常用工具：

| 工具 | 用途 | 使用方式 |
|------|------|---------|
| `torch.profiler` | PyTorch 算子级分析 | `with torch.profiler.profile(...)` |
| NVIDIA Nsight Systems | GPU 时间线分析 | `nsys profile python bench.py` |
| NVIDIA Nsight Compute | 单 kernel 深度分析 | `ncu --set full python bench.py` |
| `nvidia-smi` | 实时 GPU 状态 | `nvidia-smi -l 1` |
| `torch.cuda.memory_summary()` | 显存使用统计 | 代码中调用 |

### 5.5 常见性能问题排查

| 症状 | 可能原因 | 排查方法 |
|------|---------|---------|
| Prefill 很慢 | 未使用 FlashAttention | 检查 attention 实现 |
| Decode 很慢 | 未启用 CUDA Graph | 检查 `enforce_eager` |
| 吞吐量低 | batch size 太小 | 增大 `max_num_seqs` |
| OOM | KV Cache 占用过多 | 降低 `gpu_memory_utilization` 或 `max_model_len` |
| 首步很慢 | torch.compile 编译 | 正常现象，后续步骤会快 |
| GPU 利用率低 | CPU 调度瓶颈 | 检查是否 CUDA Graph 生效 |

## 六、高级优化方向

### 6.1 量化（Quantization）

将模型权重从 FP16（16-bit）量化到 INT8/INT4，可以：

- **减少显存占用**：7B 模型从 ~14GB 降到 ~7GB (INT8) 或 ~3.5GB (INT4)。
- **提高内存带宽利用率**：读取更少的字节完成相同的计算。
- **加速 decode**：decode 阶段是 memory-bound，减少数据量直接提速。

代价：精度可能略有下降，需要适当的量化校准。

### 6.2 Speculative Decoding（投机解码）

使用一个小模型（draft model）快速生成多个候选 token，再用大模型（target model）并行验证：

```
Draft model: 生成 k 个候选 token（很快）
Target model: 一次前向验证 k 个 token（利用 prefill 的并行性）
接受 n ≤ k 个 token
```

理论上可以将 decode 速度提升 2-3 倍，代价是额外的小模型显存和计算。

### 6.3 Continuous Batching 优化

进一步优化调度策略：

- **Iteration-level 调度**：每个 step 都可以加入新请求或移除完成的请求。
- **Preemption**：当显存不足时，暂停低优先级请求，腾出 KV Cache 给高优先级请求。
- **Chunked Prefill**：将长 prompt 分块，与 decode 交错执行，避免长 prompt 阻塞 decode。

## 七、小结

- **bench.py** 通过固定随机种子、warmup、随机长度输入输出、`ignore_eos` 等手段实现可复现的性能测试。
- LLM 推理的核心指标包括 **TTFT、TPOT、Prefill/Decode 吞吐量**。
- **CUDA Graph** 对 decode 阶段有 1.4-1.9x 加速（小 batch 效果更大）。
- **torch.compile** 通过算子融合和代码生成带来 ~1.27x 加速。
- **FlashAttention** 对长序列 prefill 有 2.5-10x+ 加速。
- **前缀缓存** 在共享前缀场景下可避免重复计算，加速 2-7x。
- nano-vllm 在核心推理性能上达到 vLLM 的 **85-95%**，考虑到代码量仅 3000 行，性价比极高。
- 性能调优的核心是 **最大化 batch size**、**启用所有优化**、**用 profiling 工具定位瓶颈**。

## 八、面试考点（含参考答案）

**1. 如何设计一个公平的 LLM 推理 benchmark？**
**答**：关键要素包括：① 固定随机种子确保可复现；② 进行 warmup 排除初始化开销；③ 使用 `ignore_eos=True` 控制输出长度；④ 输入和输出长度应覆盖不同范围（模拟真实分布）；⑤ 足够多的请求数（如 256 条）充分利用 batching；⑥ 多次重复取平均减少抖动；⑦ 确保 GPU 无其他负载。

**2. Prefill 吞吐量和 Decode 吞吐量有什么区别？分别受什么因素影响？**
**答**：Prefill 吞吐量衡量处理 prompt 的速度，受 **GPU 计算能力（FLOPS）** 影响，因为 prefill 是 compute-bound；Decode 吞吐量衡量生成新 token 的速度，受 **内存带宽和调度效率** 影响，因为 decode 是 memory-bound。两者的优化方向不同：prefill 靠 FlashAttention 提速，decode 靠 CUDA Graph + 大 batch + 量化提速。

**3. bench.py 中为什么要 `ignore_eos=True`？**
**答**：如果不忽略 EOS，不同运行之间因为采样的随机性，各序列可能在不同位置生成 EOS 导致输出长度不一致。这会影响总 token 数和时间，使性能数据不可比较。`ignore_eos=True` 强制所有序列生成到 `max_tokens`，确保工作量一致。

**4. CUDA Graph 在什么场景下加速效果最明显？**
**答**：**小 batch decode** 场景下效果最明显（1.5-2x 加速），因为此时 GPU 计算量小但 kernel launch 次数多，CPU launch overhead 占总耗时比例高。大 batch 时计算本身占主导，Graph 收益降低（但仍有 1.3-1.5x）。Prefill 阶段不适用 Graph（变长 shape）。

**5. FlashAttention 的加速原理是什么？为什么长序列效果更显著？**
**答**：FlashAttention 通过分块计算（Tiling）避免在 HBM 中存储完整的 \(N \times N\) 注意力矩阵，将 IO 复杂度从 \(O(N^2)\) 降至 \(O(N)\)。长序列时，标准 Attention 需要存储和读取的 \(N^2\) 矩阵非常大，成为严重的内存带宽瓶颈；FlashAttention 完全避免了这个开销，因此序列越长加速比越大。

**6. torch.compile 是如何加速推理的？**
**答**：`torch.compile` 通过三个层面优化：① **算子融合**：将多个小算子（如 cast + div + softmax + argmax）合并为一个大 kernel，减少 launch 开销和中间内存分配；② **内存优化**：减少临时张量的生成和回收；③ **后端代码生成**：Inductor 后端生成优化的 Triton 代码，比默认的 PyTorch eager 执行更高效。

**7. nano-vllm 和 vLLM 的性能差距主要来自哪些方面？**
**答**：主要来自：① 缺少 Chunked Prefill（无法将长 prompt 与 decode 交错执行）；② 调度策略较简化（缺少 preemption、priority 等机制）；③ 不支持量化（AWQ/GPTQ/FP8 可减少内存带宽需求）；④ 不支持 Speculative Decoding；⑤ Paged Attention 实现不如 vLLM 精细。但核心推理路径（模型前向、FlashAttention、CUDA Graph）基本对齐。

**8. 性能测试中 warmup 不充分会导致什么问题？**
**答**：会导致 **低估真实性能**。不充分的 warmup 使得测试数据包含：CUDA 上下文初始化（~100ms）、torch.compile JIT 编译（数十秒）、cuBLAS auto-tuning、内存分配器首次分配等一次性开销。这些在生产环境中只发生一次，不应计入稳态性能数据。

**9. 如何计算一个模型在给定 GPU 上的理论最大 decode 吞吐量？**
**答**：decode 阶段是 memory-bound，理论最大吞吐量 ≈ **内存带宽 / 每 token 需读取的权重字节数**。例如 7B FP16 模型在 A100-80G（2TB/s 带宽）上：每 token 需读取 ~14GB 权重，理论上限 ≈ 2000GB/s ÷ 14GB ≈ 143 tok/s（单序列）。通过 batching，多个序列共享权重读取，可以提高到数千 tok/s。

**10. 在生产环境中，除了吞吐量还需要关注哪些性能指标？**
**答**：① **TTFT（首字延迟）**：影响用户感知的响应速度；② **P99 延迟**：尾部延迟影响 SLA 达标率；③ **GPU 显存利用率**：影响可服务的最大并发数；④ **请求排队时间**：调度效率的体现；⑤ **GPU 利用率**：是否有 idle 浪费；⑥ **服务可用性**：OOM 导致的重启频率。生产环境通常需要在吞吐量和延迟之间权衡（Pareto 最优）。
