# 课程 15：张量并行（Tensor Parallelism）

本课深入剖析 nano-vllm 中的**张量并行（Tensor Parallelism, TP）**实现——它是大模型推理中最常用的多 GPU 并行策略。我们将从数学原理出发，逐类讲解 `ColumnParallelLinear`、`RowParallelLinear`、`QKVParallelLinear`、`MergedColumnParallelLinear` 和 `VocabParallelEmbedding` 的设计思想与实现细节，配合面试考点。

---

## 一、为什么需要张量并行

### 1.1 单 GPU 的瓶颈

随着大模型参数量的增长，单 GPU 面临两个瓶颈：

**显存瓶颈**：

| 模型规模 | 参数量 | FP16 权重大小 | KV Cache（2048 seq, batch=32） | 总需求 |
|---------|--------|-------------|------------------------------|--------|
| 7B | 7×10⁹ | 14 GB | ~8 GB | ~22 GB |
| 13B | 13×10⁹ | 26 GB | ~16 GB | ~42 GB |
| 70B | 70×10⁹ | 140 GB | ~80 GB | ~220 GB |

即使是 A100-80GB，也无法容纳 70B 模型的权重，更不用说 KV Cache。

**计算瓶颈**：

单 GPU 的算力有限。即使显存够用，7B 模型在单 A100 上的 Decode 吞吐量约为 2000 tokens/s，而 4 张 A100 通过张量并行可以达到 ~6000 tokens/s。

### 1.2 并行策略概览

| 策略 | 切分维度 | 通信模式 | 适用场景 |
|------|---------|---------|---------|
| **数据并行（DP）** | 按 batch 切分 | AllReduce 梯度 | 训练 |
| **张量并行（TP）** | 按权重矩阵切分 | AllReduce 激活值 | 推理（同节点） |
| **流水线并行（PP）** | 按层切分 | 点对点发送激活值 | 训练 + 推理（跨节点） |
| **序列并行（SP）** | 按序列长度切分 | AllGather/ReduceScatter | 长序列场景 |

### 1.3 为什么推理首选张量并行

推理场景下，TP 相比其他策略有独特优势：

1. **低延迟**：所有 GPU 同时计算，一层的计算时间几乎是 \( 1/N \)（N 为 GPU 数）。PP 则需要按层串行，延迟无法降低。
2. **高利用率**：每个 GPU 始终有活干。PP 在推理中存在"气泡"（等待上一层输出）。
3. **AllReduce 高效**：同一节点内的 GPU 通过 NVLink 连接，AllReduce 带宽高达 600 GB/s。
4. **实现相对简单**：只需修改 Linear 层的权重划分方式。

### 1.4 张量并行的核心思想

张量并行的核心是**矩阵分块乘法**。对于线性层 \( Y = XW + b \)，将权重矩阵 \( W \) 沿某个维度切分到多个 GPU 上，每个 GPU 只计算部分结果，最后通过通信汇总。

```
单 GPU:     Y = X × W           [M×K] × [K×N] = [M×N]

2-GPU TP:   Y₀ = X × W₀        [M×K] × [K×N/2] = [M×N/2]   ← GPU 0
            Y₁ = X × W₁        [M×K] × [K×N/2] = [M×N/2]   ← GPU 1
            Y = [Y₀, Y₁]       拼接得到 [M×N]
```

关键问题是：**沿哪个维度切分**，以及**什么时候需要通信**。

## 二、列并行（ColumnParallelLinear）

### 2.1 数学原理

将权重矩阵 \( W \in \mathbb{R}^{K \times N} \) **按列（output 维度）切分**为 \( N_{\text{tp}} \) 份：

\[
W = [W_0, W_1, \ldots, W_{N_{\text{tp}}-1}], \quad W_i \in \mathbb{R}^{K \times (N/N_{\text{tp}})}
\]

每个 GPU 计算：

\[
Y_i = X \cdot W_i, \quad Y_i \in \mathbb{R}^{M \times (N/N_{\text{tp}})}
\]

**结果是完整输出的一个分片**——每个 GPU 持有 \( N/N_{\text{tp}} \) 列输出。

```
输入 X:      所有 GPU 相同  [M × K]
权重 W₀:     GPU 0         [K × N/2]
权重 W₁:     GPU 1         [K × N/2]
输出 Y₀:     GPU 0         [M × N/2]  ← 前 N/2 列
输出 Y₁:     GPU 1         [M × N/2]  ← 后 N/2 列
```

### 2.2 关键特性

1. **输入不需要通信**：每个 GPU 接收相同的完整输入 \( X \)
2. **输出是分片的**：每个 GPU 只持有部分输出列
3. **后续层如果也是列并行**，可以直接将分片输出作为输入，无需通信
4. **如果后续层需要完整输入**（如 RowParallelLinear），则需要通信

### 2.3 nano-vllm 实现

```python
class ColumnParallelLinear(LinearBase):
    def __init__(self, input_size, output_size, bias=False):
        # output_size 除以 tp_size：每个 GPU 只存 1/tp_size 的输出列
        super().__init__(input_size, divide(output_size, tp_size), bias, tp_dim=0)

    def weight_loader(self, param, loaded_weight):
        # 从完整权重中取出本 GPU 对应的列分片
        shard_size = param_data.size(self.tp_dim)
        start_idx = self.tp_rank * shard_size
        loaded_weight = loaded_weight.narrow(self.tp_dim, start_idx, shard_size)
        param_data.copy_(loaded_weight)
```

#### 构造函数解读

```python
super().__init__(input_size, divide(output_size, tp_size), bias, tp_dim=0)
```

- `input_size`：保持不变（每个 GPU 接收完整输入）
- `output_size / tp_size`：每个 GPU 只输出 \( 1/N_{\text{tp}} \) 的列
- `tp_dim=0`：切分维度是权重的第 0 维（即输出维度，因为 PyTorch 的 Linear 权重形状是 `[output, input]`）

#### weight_loader 解读

`weight_loader` 在模型加载时被调用，负责从 HuggingFace 的完整权重中提取本 GPU 对应的分片：

```python
shard_size = param_data.size(self.tp_dim)   # 本 GPU 的分片大小 = output_size / tp_size
start_idx = self.tp_rank * shard_size        # 起始索引 = rank × 分片大小
loaded_weight = loaded_weight.narrow(self.tp_dim, start_idx, shard_size)
```

`narrow` 是 PyTorch 的切片操作，类似于 `loaded_weight[start_idx:start_idx+shard_size]`（但沿指定维度）。

### 2.4 前向传播

```python
def forward(self, x):
    return F.linear(x, self.weight, self.bias)
```

列并行的前向传播就是标准的矩阵乘法——**不需要通信**。通信在后续的 RowParallelLinear 中发生。

### 2.5 列并行在 Transformer 中的应用

在 Transformer 中，以下层使用列并行：

- **Q/K/V 投影**（`QKVParallelLinear`）：每个 GPU 持有部分注意力头的 Q/K/V
- **FFN 的第一个线性层**（gate_proj, up_proj）：每个 GPU 持有部分中间维度
- **LM Head**（词表映射）：每个 GPU 持有部分词表

## 三、行并行（RowParallelLinear）

### 3.1 数学原理

将权重矩阵 \( W \in \mathbb{R}^{K \times N} \) **按行（input 维度）切分**：

\[
W = \begin{bmatrix} W_0 \\ W_1 \\ \vdots \\ W_{N_{\text{tp}}-1} \end{bmatrix}, \quad W_i \in \mathbb{R}^{(K/N_{\text{tp}}) \times N}
\]

输入 \( X \) 也**对应切分**（通常是上一个 ColumnParallelLinear 的分片输出）：

\[
X = [X_0, X_1, \ldots, X_{N_{\text{tp}}-1}], \quad X_i \in \mathbb{R}^{M \times (K/N_{\text{tp}})}
\]

每个 GPU 计算局部结果：

\[
Y_i = X_i \cdot W_i, \quad Y_i \in \mathbb{R}^{M \times N}
\]

**关键**：每个 \( Y_i \) 的形状都是 \( [M \times N] \)（完整的输出形状），但只是**部分和**。完整输出需要：

\[
Y = \sum_{i=0}^{N_{\text{tp}}-1} Y_i = X_0 W_0 + X_1 W_1 + \cdots
\]

这就是 **AllReduce** 操作——将所有 GPU 的部分和汇总。

### 3.2 为什么行并行需要 AllReduce

回顾矩阵乘法的分块形式：

\[
Y = X \cdot W = [X_0, X_1] \cdot \begin{bmatrix} W_0 \\ W_1 \end{bmatrix} = X_0 W_0 + X_1 W_1
\]

每个 GPU 只计算了 \( X_i W_i \)，是最终结果的一部分。需要通过 AllReduce（sum）将所有 GPU 的结果加在一起才能得到正确输出。

### 3.3 nano-vllm 实现

```python
class RowParallelLinear(LinearBase):
    def __init__(self, input_size, output_size, bias=False):
        # input_size 除以 tp_size：每个 GPU 只接收 1/tp_size 的输入
        super().__init__(divide(input_size, tp_size), output_size, bias, tp_dim=1)

    def forward(self, x):
        # 1. 局部矩阵乘法
        y = F.linear(x, self.weight, self.bias if self.tp_rank == 0 else None)
        # 2. AllReduce 求和
        if self.tp_size > 1:
            dist.all_reduce(y)
        return y
```

#### 构造函数解读

```python
super().__init__(divide(input_size, tp_size), output_size, bias, tp_dim=1)
```

- `input_size / tp_size`：每个 GPU 只接收分片输入
- `output_size`：保持完整（每个 GPU 输出完整形状，但是部分和）
- `tp_dim=1`：切分维度是权重的第 1 维（input 维度）

#### 前向传播解读

```python
y = F.linear(x, self.weight, self.bias if self.tp_rank == 0 else None)
```

**Bias 只在 rank 0 加**：因为 AllReduce 会对所有 GPU 的结果求和。如果每个 GPU 都加 bias，最终结果会被加了 \( N_{\text{tp}} \) 次 bias。所以只有 rank 0 加 bias，AllReduce 后结果正确。

```python
if self.tp_size > 1:
    dist.all_reduce(y)
```

`dist.all_reduce(y)` 是 NCCL 的 AllReduce 操作，默认求和。所有 GPU 的 `y` 被汇总为：

\[
y_{\text{final}} = y_0 + y_1 + \cdots + y_{N_{\text{tp}}-1}
\]

操作完成后，**每个 GPU 都持有相同的完整输出**。

### 3.4 行并行在 Transformer 中的应用

- **Attention 的输出投影**（o_proj）：接收列并行的注意力输出分片，通过行并行 + AllReduce 恢复完整输出
- **FFN 的第二个线性层**（down_proj）：接收列并行的中间结果分片，通过行并行 + AllReduce 恢复完整输出

## 四、列并行 + 行并行的配对模式

### 4.1 配对原理

在 Transformer 中，列并行和行并行总是**成对出现**：

```
Column Parallel                    Row Parallel
(切分输出维度)                      (切分输入维度 + AllReduce)

    X [M×K]                         X₀ [M×K/2] (GPU 0)
    ↓                               X₁ [M×K/2] (GPU 1)
    W₀ [K×N/2]  (GPU 0)            ↓
    W₁ [K×N/2]  (GPU 1)            W₀ [K/2×N]  (GPU 0)
    ↓                               W₁ [K/2×N]  (GPU 1)
    Y₀ [M×N/2]  (GPU 0)            ↓
    Y₁ [M×N/2]  (GPU 1)            Y₀ [M×N]    (GPU 0)  ← 部分和
                                    Y₁ [M×N]    (GPU 1)  ← 部分和
    无需通信                         ↓ AllReduce
                                    Y [M×N]     (所有 GPU 相同)
```

### 4.2 在 Transformer 中的应用

#### Attention 层

```
输入 X [M×D]（所有 GPU 相同）
    ↓
QKVParallelLinear（列并行）
    → Q₀, K₀, V₀ [M×D/TP]  (GPU 0)
    → Q₁, K₁, V₁ [M×D/TP]  (GPU 1)
    ↓
Attention 计算（每个 GPU 独立计算自己的头）
    → O₀ [M×D/TP]  (GPU 0)
    → O₁ [M×D/TP]  (GPU 1)
    ↓
o_proj（行并行 + AllReduce）
    → Y [M×D]（所有 GPU 相同）
```

#### FFN 层

```
输入 X [M×D]（所有 GPU 相同）
    ↓
gate_proj, up_proj（列并行 / MergedColumnParallelLinear）
    → gate₀, up₀ [M×D_ff/TP]  (GPU 0)
    → gate₁, up₁ [M×D_ff/TP]  (GPU 1)
    ↓
激活函数 + 逐元素乘法（每个 GPU 独立）
    → H₀ [M×D_ff/TP]  (GPU 0)
    → H₁ [M×D_ff/TP]  (GPU 1)
    ↓
down_proj（行并行 + AllReduce）
    → Y [M×D]（所有 GPU 相同）
```

### 4.3 通信量分析

每层 Transformer 需要 **2 次 AllReduce**（一次来自 Attention 的 o_proj，一次来自 FFN 的 down_proj）。

单次 AllReduce 的通信量：
\[
\text{通信量} = 2 \times (N_{\text{tp}} - 1) / N_{\text{tp}} \times M \times D \times \text{sizeof(dtype)}
\]

对于 M=1（Decode 单 token）、D=4096、FP16、TP=4：
\[
= 2 \times 3/4 \times 1 \times 4096 \times 2 = 12 \text{ KB}
\]

**非常小！** 这就是为什么 TP 在同节点内（NVLink）效率极高。

对于 M=1000（Prefill 长序列）：
\[
= 2 \times 3/4 \times 1000 \times 4096 \times 2 = 12 \text{ MB}
\]

仍然可以接受（NVLink 的单向带宽 300+ GB/s，传输 12 MB 仅需 ~40 μs）。

## 五、QKVParallelLinear 的特殊处理

### 5.1 为什么需要特殊处理

在标准 Transformer 中，Q、K、V 分别由三个独立的线性层生成。但在实际实现中，为了减少 kernel launch 开销，通常将 Q、K、V 的权重**合并为一个大矩阵**，一次矩阵乘法同时得到 Q、K、V。

当使用 GQA（Grouped Query Attention）时，Q 的头数（如 32）和 KV 的头数（如 4）不同，权重矩阵的 Q 部分和 KV 部分大小不一致。加上张量并行的切分，权重加载时需要特殊的分片逻辑。

### 5.2 合并权重的结构

```
QKV 合并权重 W_qkv: [hidden_size, (num_heads + 2×num_kv_heads) × head_dim]

拆分为：
├── Q 部分：[hidden_size, num_heads × head_dim]
├── K 部分：[hidden_size, num_kv_heads × head_dim]
└── V 部分：[hidden_size, num_kv_heads × head_dim]
```

### 5.3 nano-vllm 实现

```python
class QKVParallelLinear(ColumnParallelLinear):
    def __init__(self, hidden_size, head_size, total_num_heads,
                 total_num_kv_heads=None):
        total_num_kv_heads = total_num_kv_heads or total_num_heads
        self.head_size = head_size
        self.num_heads = divide(total_num_heads, tp_size)
        self.num_kv_heads = divide(total_num_kv_heads, tp_size)

        # 总输出大小 = (Q头数 + 2×KV头数) × 头维度
        output_size = (total_num_heads + 2 * total_num_kv_heads) * head_size
        super().__init__(hidden_size, output_size)
```

#### 构造函数解读

```python
self.num_heads = divide(total_num_heads, tp_size)      # 每 GPU 的 Q 头数
self.num_kv_heads = divide(total_num_kv_heads, tp_size) # 每 GPU 的 KV 头数
```

以 Qwen3-7B 为例（total_num_heads=32, total_num_kv_heads=4, tp_size=2）：
- 每 GPU 的 Q 头数：32 / 2 = 16
- 每 GPU 的 KV 头数：4 / 2 = 2

### 5.4 weight_loader 的特殊逻辑

QKVParallelLinear 的 `weight_loader` 需要区别处理 Q 和 KV 的权重分片：

```python
def weight_loader(self, param, loaded_weight, weight_name):
    if "q_proj" in weight_name:
        # Q 权重按 num_heads 切分
        shard_size = self.num_heads * self.head_size
        start_idx = self.tp_rank * shard_size
        offset = 0
    elif "k_proj" in weight_name:
        # K 权重按 num_kv_heads 切分
        shard_size = self.num_kv_heads * self.head_size
        start_idx = self.tp_rank * shard_size
        offset = self.num_heads * self.head_size  # Q 之后
    elif "v_proj" in weight_name:
        # V 权重按 num_kv_heads 切分
        shard_size = self.num_kv_heads * self.head_size
        start_idx = self.tp_rank * shard_size
        offset = (self.num_heads + self.num_kv_heads) * self.head_size  # Q+K 之后

    loaded_weight = loaded_weight.narrow(0, start_idx, shard_size)
    param_data[offset:offset+shard_size].copy_(loaded_weight)
```

每个 GPU 的权重布局：

```
GPU 的 QKV 权重：[Q_shard | K_shard | V_shard]
                  ↑         ↑         ↑
           num_heads×D  num_kv×D  num_kv×D
           (16×128)     (2×128)   (2×128)
```

### 5.5 前向传播后的拆分

前向传播得到 QKV 的拼接结果后，需要拆分为 Q、K、V 三个张量：

```python
# attention.py
qkv = self.qkv_proj(x)  # [M, (num_heads + 2×num_kv_heads) × head_dim]
q, k, v = qkv.split([
    self.num_heads * self.head_dim,
    self.num_kv_heads * self.head_dim,
    self.num_kv_heads * self.head_dim
], dim=-1)
```

## 六、MergedColumnParallelLinear

### 6.1 设计动机

在 Transformer FFN 中，通常有两个列并行的线性层需要同时计算（如 Qwen3 的 `gate_proj` 和 `up_proj`）。为了减少 kernel launch 开销，将它们合并为一个大矩阵乘法：

```
分开计算（2 次 kernel launch）：
  gate = x @ W_gate    [M×D] × [D×D_ff/TP]
  up   = x @ W_up      [M×D] × [D×D_ff/TP]

合并计算（1 次 kernel launch）：
  gate_up = x @ [W_gate, W_up]   [M×D] × [D×2D_ff/TP]
```

### 6.2 实现

```python
class MergedColumnParallelLinear(ColumnParallelLinear):
    def __init__(self, input_size, output_sizes, bias=False):
        # output_sizes 是多个列并行层的输出大小列表
        # 例如 [intermediate_size, intermediate_size]
        self.output_sizes = output_sizes
        total_output_size = sum(output_sizes)
        super().__init__(input_size, total_output_size, bias)
```

### 6.3 weight_loader 的特殊逻辑

加载时需要将每个子层的权重放到合并矩阵的正确位置：

```python
def weight_loader(self, param, loaded_weight, weight_name):
    # 确定是 gate_proj 还是 up_proj
    if "gate_proj" in weight_name:
        idx = 0
    elif "up_proj" in weight_name:
        idx = 1

    # 计算在合并权重中的偏移
    shard_size = divide(self.output_sizes[idx], tp_size)
    offset = sum(divide(s, tp_size) for s in self.output_sizes[:idx])
    start_idx = self.tp_rank * shard_size

    loaded_weight = loaded_weight.narrow(0, start_idx, shard_size)
    param_data[offset:offset+shard_size].copy_(loaded_weight)
```

合并后每个 GPU 的权重布局：

```
[gate_shard | up_shard]
 D_ff/TP      D_ff/TP
```

前向传播后，通过 `split` 拆回：

```python
gate, up = merged_output.split(
    [intermediate_size // tp_size, intermediate_size // tp_size], dim=-1
)
output = gate * F.silu(up)  # SwiGLU 激活
```

## 七、词表并行（VocabParallelEmbedding）

### 7.1 问题

Embedding 层将 token ID 映射为 embedding 向量。词表大小通常很大（如 152,064），embedding 矩阵占用大量显存。

标准 Embedding：
```
W_embed: [vocab_size, hidden_size] = [152064, 4096]
大小 = 152064 × 4096 × 2 bytes ≈ 1.2 GB
```

### 7.2 并行策略

将词表**按行切分**到多个 GPU：

```
GPU 0: W_embed[0:76032, :]           ← 负责 token ID 0~76031
GPU 1: W_embed[76032:152064, :]      ← 负责 token ID 76032~152063
```

每个 GPU 只查找自己负责范围内的 token，对于范围外的 token 返回零向量：

```python
# 假设 tp_rank=0, vocab_start=0, vocab_end=76032
mask = (input_ids >= vocab_start) & (input_ids < vocab_end)
local_ids = input_ids - vocab_start   # 转换为本地索引
local_ids[~mask] = 0                   # 范围外的设为 0

embeddings = F.embedding(local_ids, self.weight)  # 本地查找
embeddings[~mask] = 0                              # 范围外的清零
```

然后通过 **AllReduce（sum）** 汇总：

```python
if self.tp_size > 1:
    dist.all_reduce(embeddings)
```

由于每个 token 只在一个 GPU 上有非零 embedding，AllReduce 后每个 GPU 都持有完整的、正确的 embedding。

### 7.3 LM Head 的词表并行

LM Head 是 Embedding 的逆操作（通常共享权重），将隐藏状态映射回词表大小的 logits：

```
logits = hidden_states @ W_embed.T    [M×D] × [D×vocab_size] = [M×vocab_size]
```

在张量并行中，每个 GPU 计算部分词表的 logits：

```python
local_logits = F.linear(hidden_states, self.weight)  # [M, vocab_size/tp_size]
```

然后通过 **AllGather** 拼接：

```python
if self.tp_size > 1:
    all_logits = [torch.empty_like(local_logits) for _ in range(tp_size)]
    dist.all_gather(all_logits, local_logits)
    logits = torch.cat(all_logits, dim=-1)  # [M, vocab_size]
```

AllGather 后，每个 GPU 都持有完整的 logits，可以进行采样。

## 八、NCCL 通信初始化

### 8.1 进程组初始化

```python
# model_runner.py
dist.init_process_group(
    backend="nccl",                  # 使用 NCCL 后端
    init_method="tcp://localhost:2333",  # 集合点地址
    world_size=self.world_size,      # GPU 总数
    rank=rank                        # 当前进程的编号
)
```

**NCCL（NVIDIA Collective Communication Library）** 是 NVIDIA 提供的 GPU 间高性能集合通信库，支持 AllReduce、AllGather、ReduceScatter 等操作。

**init_method**：指定进程间的集合点（rendezvous）。所有进程通过这个地址互相发现。`tcp://localhost:2333` 表示使用 TCP 方式，在本机 2333 端口建立连接。

### 8.2 多进程启动

nano-vllm 使用 Python 的 `multiprocessing` 启动多个进程：

```python
# llm_engine.py
if config.tensor_parallel_size > 1:
    processes = []
    event = Event()
    for rank in range(1, config.tensor_parallel_size):
        p = Process(target=ModelRunner, args=(config, rank, event))
        p.start()
        processes.append(p)
    # rank 0 在主进程中运行
    self.model_runner = ModelRunner(config, rank=0, event=event)
```

- Rank 0 在主进程中运行，负责调度和采样
- Rank 1~N 在子进程中运行，创建 ModelRunner 后进入 `loop()` 等待指令

### 8.3 NCCL 通信的硬件基础

| 互联方式 | 带宽 | 延迟 | 适用场景 |
|---------|------|------|---------|
| NVLink (A100) | 600 GB/s | ~1 μs | 同节点 GPU 间 |
| NVLink (H100) | 900 GB/s | ~1 μs | 同节点 GPU 间 |
| PCIe 4.0 x16 | 32 GB/s | ~2 μs | 同节点 CPU-GPU |
| InfiniBand HDR | 200 Gb/s (25 GB/s) | ~1-3 μs | 跨节点 |
| Ethernet 100G | 12.5 GB/s | ~5-10 μs | 跨节点 |

nano-vllm 只支持单节点多 GPU（通过 NVLink），因此 AllReduce 的通信延迟极低。

### 8.4 AllReduce 的实现原理

NCCL 的 AllReduce 使用 **Ring AllReduce** 算法：

```
假设 4 个 GPU，每个有数据 [A₀, A₁, A₂, A₃]

Step 1: Reduce-Scatter
  GPU 0 → GPU 1 → GPU 2 → GPU 3 → GPU 0 (环形传递)
  3 步后，每个 GPU 持有 1/4 的完整和

Step 2: All-Gather
  GPU 0 → GPU 1 → GPU 2 → GPU 3 → GPU 0 (环形传递)
  3 步后，每个 GPU 持有完整的和

总通信量：2 × (N-1)/N × 数据量 ≈ 2 × 数据量（当 N 大时）
```

Ring AllReduce 的通信量与 GPU 数量**近乎无关**（随 N 增大趋近常数），因此张量并行的通信开销可以很好地扩展。

## 九、完整的张量并行数据流示例

### 9.1 单层 Transformer 的数据流（TP=2）

```
输入 X: [M×D] (两个 GPU 相同)
│
├── GPU 0                              ├── GPU 1
│   QKV_proj (ColumnParallel)          │   QKV_proj (ColumnParallel)
│   W_q₀: [D, D/2]                    │   W_q₁: [D, D/2]
│   W_k₀: [D, D_kv/2]                 │   W_k₁: [D, D_kv/2]
│   W_v₀: [D, D_kv/2]                 │   W_v₁: [D, D_kv/2]
│   → Q₀, K₀, V₀                      │   → Q₁, K₁, V₁
│                                       │
│   Attention(Q₀, K₀, V₀)             │   Attention(Q₁, K₁, V₁)
│   → O₀: [M, D/2]                    │   → O₁: [M, D/2]
│                                       │
│   o_proj (RowParallel)               │   o_proj (RowParallel)
│   W_o₀: [D/2, D]                    │   W_o₁: [D/2, D]
│   → Y₀: [M, D] (部分和)              │   → Y₁: [M, D] (部分和)
│                                       │
│        ←──── AllReduce (sum) ────→
│   Y_attn: [M, D] (完整)              │   Y_attn: [M, D] (完整)
│                                       │
│   + Residual + LayerNorm              │   + Residual + LayerNorm
│   → X': [M, D]                       │   → X': [M, D]
│                                       │
│   gate_up_proj (MergedColumnParallel) │   gate_up_proj (MergedColumnParallel)
│   W_gate₀: [D, D_ff/2]              │   W_gate₁: [D, D_ff/2]
│   W_up₀: [D, D_ff/2]                │   W_up₁: [D, D_ff/2]
│   → gate₀, up₀                       │   → gate₁, up₁
│   → SiLU(gate₀) × up₀               │   → SiLU(gate₁) × up₁
│   → H₀: [M, D_ff/2]                │   → H₁: [M, D_ff/2]
│                                       │
│   down_proj (RowParallel)            │   down_proj (RowParallel)
│   W_down₀: [D_ff/2, D]              │   W_down₁: [D_ff/2, D]
│   → Z₀: [M, D] (部分和)              │   → Z₁: [M, D] (部分和)
│                                       │
│        ←──── AllReduce (sum) ────→
│   Y_ffn: [M, D] (完整)               │   Y_ffn: [M, D] (完整)
│                                       │
│   + Residual                          │   + Residual
│   → 下一层的输入                       │   → 下一层的输入
```

**关键观察**：整层只需要 **2 次 AllReduce**（Attention 一次，FFN 一次），其他计算完全并行，无需通信。

## 十、面试高频考点

### Q1：请解释张量并行中 ColumnParallelLinear 和 RowParallelLinear 的区别。

**参考答案**：

| 维度 | ColumnParallelLinear | RowParallelLinear |
|------|---------------------|-------------------|
| 切分方式 | 权重按列（output）切分 | 权重按行（input）切分 |
| 输入要求 | 完整输入（所有 GPU 相同） | 分片输入（每 GPU 不同） |
| 输出形式 | 分片输出（每 GPU 持有部分列） | 部分和（每 GPU 持有完整形状但不完整值） |
| 通信需求 | 不需要 | 需要 AllReduce（sum） |
| 使用位置 | Q/K/V 投影、FFN gate/up | Attention o_proj、FFN down_proj |

两者总是配对使用：ColumnParallel 的分片输出直接作为 RowParallel 的分片输入，RowParallel 通过 AllReduce 恢复完整输出。

### Q2：为什么 RowParallelLinear 中 bias 只在 rank 0 添加？

**参考答案**：

因为 AllReduce 是求和操作。如果每个 GPU 都加 bias，AllReduce 后 bias 会被加了 tp_size 次。正确做法是只在一个 GPU（通常 rank 0）上加 bias，AllReduce 求和后结果自然包含一次 bias。

数学表达：
- 错误：\( \sum_i (X_i W_i + b) = XW + N_{\text{tp}} \cdot b \)
- 正确：\( \sum_i X_i W_i + b = XW + b \)（只有 rank 0 加 b）

### Q3：张量并行每层需要几次通信？通信量是多少？

**参考答案**：

每层 Transformer 需要 2 次 AllReduce：
1. Attention 的 o_proj 输出
2. FFN 的 down_proj 输出

Ring AllReduce 的通信量 = \( 2 \times (N_{\text{tp}} - 1) / N_{\text{tp}} \times \text{数据量} \)

Decode 阶段（单 token）：通信量 ≈ 2 × 2 × D × 2 bytes。对于 D=4096，约 32 KB/层。总共 32 层 ≈ 1 MB。在 NVLink（600 GB/s）下仅需 ~2 μs，远小于计算时间。

### Q4：请比较 TP、PP 和 DP 三种并行策略的优缺点。

**参考答案**：

| | 张量并行（TP） | 流水线并行（PP） | 数据并行（DP） |
|--|-------------|-------------|-------------|
| 切分维度 | 权重矩阵 | 层 | Batch |
| 通信方式 | AllReduce | 点对点 | AllReduce 梯度 |
| 通信频率 | 每层 2 次 | 每微批 1 次 | 每步 1 次 |
| 延迟 | 低（并行计算） | 中（串行流水线） | 不降低 |
| 吞吐量 | 近线性扩展 | 有气泡 | 线性扩展 |
| 适用场景 | 同节点推理 | 跨节点推理 | 训练 |
| 实现复杂度 | 中 | 高 | 低 |

推理场景通常优先选择 TP（延迟最低）。当 GPU 数超过单节点限制时，结合 PP 做跨节点并行。DP 主要用于训练。

### Q5：GQA 对张量并行有什么影响？

**参考答案**：

GQA（Grouped Query Attention）中 Q 的头数远大于 KV 的头数（如 Q=32, KV=4）。在张量并行中：
- Q 按 `total_num_heads / tp_size` 切分
- KV 按 `total_num_kv_heads / tp_size` 切分

这要求 `total_num_kv_heads` 必须能被 `tp_size` 整除。例如 KV=4 时，tp_size 只能是 1, 2, 4。如果 tp_size=8，则无法切分 4 个 KV 头。

`QKVParallelLinear` 的 `weight_loader` 通过分别处理 Q 和 KV 的分片偏移量来正确加载权重。

### Q6：nano-vllm 使用什么通信后端？为什么选择 NCCL？

**参考答案**：

nano-vllm 使用 NCCL（NVIDIA Collective Communication Library）作为通信后端。选择原因：
1. **专为 NVIDIA GPU 优化**：自动利用 NVLink、NVSwitch 等硬件特性
2. **支持所有集合通信原语**：AllReduce、AllGather、ReduceScatter 等
3. **GPU Direct**：数据直接在 GPU 间传输，不经过 CPU
4. **Ring 和 Tree 算法自适应**：根据拓扑和数据量自动选择最优算法
5. **与 PyTorch 深度集成**：`torch.distributed` 原生支持 NCCL 后端

### Q7：如果模型参数太大，单节点放不下怎么办？

**参考答案**：

需要结合多种并行策略：

1. **TP + PP**：节点内用 TP（利用 NVLink），节点间用 PP（利用 InfiniBand）
2. **TP + PP + DP**：对于训练场景，还可以加入 DP 提高吞吐量
3. **模型量化**：将权重从 FP16 量化为 INT8/INT4，减少显存和通信量
4. **Offloading**：将不活跃的层参数暂存到 CPU 内存

nano-vllm 只支持单节点 TP，不支持 PP 或跨节点并行。vLLM 和 DeepSpeed Inference 支持更复杂的并行组合。

## 十一、小结

| 要点 | 内容 |
|------|------|
| **为什么需要 TP** | 单 GPU 显存/算力不足，TP 可线性扩展 |
| **列并行** | 按输出维度切分权重，无需通信，输出为分片 |
| **行并行** | 按输入维度切分权重，输出为部分和，需要 AllReduce |
| **配对模式** | Column → Row + AllReduce，每层 2 次通信 |
| **QKVParallelLinear** | Q 和 KV 头数不同，需特殊的分片和加载逻辑 |
| **MergedColumnParallel** | 合并 gate_proj + up_proj 减少 kernel launch |
| **词表并行** | Embedding 按词表切分 + AllReduce，LM Head 按词表切分 + AllGather |
| **NCCL** | GPU 间高性能通信，Ring AllReduce 通信量与 GPU 数近乎无关 |

**核心记忆口诀**：

> 列并行**分输出**，行并行**分输入**；
> Column **不通信**，Row **AllReduce**；
> 配对使用**一进一出**，每层只需**两次通信**；
> QKV **头数不同分开切**，Merged **合并省 launch**；
> NCCL **NVLink 快**，TP 推理**首选它**。
