## 一、KV Cache（8 题）

### 1. 什么是 KV Cache？为什么需要它？

**问题：** 什么是 KV Cache？为什么需要它？

**详细答案：**  
自注意力在解码阶段每一步只新增一个 query token，但 key/value 依赖于**已生成的全部前缀位置**。若每步都对整段序列重算 K、V，计算量随长度平方增长且大量重复。KV Cache 将历史位置的 **K、V 张量持久化在显存**中，新步只算当前步的 Q 并与缓存的 K、V 做注意力，从而把每步复杂度从「与全长相关的高阶」降为「与当前步相关、与缓存读取相关」。  
在 nano-vllm 中，每层 `Attention` 持有 `k_cache`/`v_cache` 视图，前向时先把本步算出的 k、v **写入**对应 slot，再调用 FlashAttention 接口读缓存做注意力。

**结合 nano-vllm 源码：**  
`Attention.forward` 在存在 cache 时先 `store_kvcache(k, v, k_cache, v_cache, context.slot_mapping)`，再根据 `context.is_prefill` 走 varlen 或 `flash_attn_with_kvcache`（见 `layers/attention.py`）。

**面试加分技巧：**  
先答「解码复用历史 K/V、避免重复投影与注意力重算」，再补「与PagedAttention/块管理结合才能实现多序列高效共卡」，最后点 nano-vllm 的 **slot 映射写入** 这一落地细节。

### 2. KV Cache 的显存计算公式是什么？

**问题：** KV Cache 的显存计算公式是什么？

**详细答案：**  
单层、单序列的朴素估算：对每层存储 **K 与 V** 两份，形状与 `(seq_len, num_kv_heads, head_dim)` 相关，总字节约 `2 × num_layers × seq_len × num_kv_heads × head_dim × sizeof(dtype)`。实际系统中 seq_len 被 **分块**（block），总占用按「块数 × 每块字节」计算；若使用张量并行，**每卡 KV 头数**为 `num_kv_heads // tp_size`。  
nano-vllm 在分配时不是按单一序列，而是按 **全局块池**：先算单块字节 `block_bytes`，再用剩余可用显存除以 `block_bytes` 得到 `num_kvcache_blocks`。

**结合 nano-vllm 源码：**  

```109:111:nano-vllm-main/nanovllm/engine/model_runner.py
        block_bytes = 2 * hf_config.num_hidden_layers * self.block_size * num_kv_heads * head_dim * hf_config.torch_dtype.itemsize
        config.num_kvcache_blocks = int(total * config.gpu_memory_utilization - used - peak + current) // block_bytes
```

其中 `num_kv_heads` 已除以 `world_size`（TP）。

**面试加分技巧：**  
写出 **2×L×…**（K+V），说明 **TP 按头切分**，并过渡到 **块化后按 block_bytes 乘块数**，体现从教科书公式到工程实现的跨越。

### 3. Prefill 和 Decode 阶段 KV Cache 的使用有什么区别？

**问题：** Prefill 和 Decode 阶段 KV Cache 的使用有什么区别？

**详细答案：**  
Prefill 阶段通常一次性处理 prompt 中大量 token（或 prefix 未命中部分），需要为**多个 query 位置**并行写入 K/V，注意力是「长序列对长序列」的因果模式；Decode 阶段每序列往往只追加 **1 个 token**，注意力退化为 batch 内多序列、每序列长度为 context+1 的 **增量解码**。  
数据准备上前者构造 `cu_seqlens_q/k`、可能带 prefix cache 的 `block_tables`；后者构造 `context_lens`、`slot_mapping` 指向**当前写入槽位**。nano-vllm 用全局 `Context` 区分两种模式。

**结合 nano-vllm 源码：**  
`ModelRunner.prepare_prefill` 设置 `set_context(True, cu_seqlens_q, ...)`；`prepare_decode` 设置 `set_context(False, slot_mapping=..., context_lens=..., block_tables=...)`（`model_runner.py`）。

**面试加分技巧：**  
一句话对比「prefill 算得多、decode 算得少但步数多」，并强调 **FlashAttention API 不同**（varlen vs with_kvcache）与 **调度上 prefill 优先** 的工程原因。

### 4. 什么是 Prefix Cache（前缀缓存）？nano-vllm 如何实现？

**问题：** 什么是 Prefix Cache（前缀缓存）？nano-vllm 如何实现？

**详细答案：**  
Prefix Cache 指多条请求或多次调用共享**相同 token 前缀**时，不必重复计算前缀的 KV，可直接复用物理块中的缓存，从而降低首包延迟与算力。实现要点是 **前缀与物理块的哈希映射** 与 **引用计数**。  
nano-vllm 用 `xxhash` 对块内 token 做哈希，并**链式**传入上一块哈希作为前缀，使相同前缀链可命中；命中时增加 `ref_count` 并推进 `num_cached_tokens`。

**结合 nano-vllm 源码：**  
`BlockManager.compute_hash` 使用 `prefix.to_bytes` + token 字节；`allocate` 中若 `hash_to_block_id` 命中且 `token_ids` 一致则复用并 `num_cached_tokens += block_size`（`block_manager.py`）。Prefill 若存在前缀缓存，`prepare_prefill` 会准备 `block_tables`（`model_runner.py` 中 `cu_seqlens_k[-1] > cu_seqlens_q[-1]` 分支）。

**面试加分技巧：**  
说明 **块级粒度**、**哈希冲突时的 token 比对**、以及 **与 PagedAttention block table 协同** 三点。

### 5. nano-vllm 的 KV Cache 张量形状是什么？为什么这么设计？

**问题：** nano-vllm 的 KV Cache 张量形状是什么？为什么这么设计？

**详细答案：**  
整体形状为 **`[2, num_layers, num_blocks, block_size, num_kv_heads, head_dim]`**：维 0 为 K/V；维 1 为层；维 2 为全局物理块；块内再按 `block_size` 展开放 token；后两维为 GQA/MHA 的 KV 头与头维。这样 **同一物理块在显存中连续**，便于用 **slot = block_id * block_size + offset** 一维索引，并与 FlashAttention 的 `block_table` 语义一致。  
张量并行下 `num_kv_heads` 为**本卡头数**。

**结合 nano-vllm 源码：**  

```112:118:nano-vllm-main/nanovllm/engine/model_runner.py
        self.kv_cache = torch.empty(2, hf_config.num_hidden_layers, config.num_kvcache_blocks, self.block_size, num_kv_heads, head_dim)
        layer_id = 0
        for module in self.model.modules():
            if hasattr(module, "k_cache") and hasattr(module, "v_cache"):
                module.k_cache = self.kv_cache[0, layer_id]
                module.v_cache = self.kv_cache[1, layer_id]
```

**面试加分技巧：**  
先背形状，再解释 **「物理块池 + 每 attention 层切片」** 与 vLLM 论文中PagedAttention存储一致，体现读过源码。

### 6. 如何动态计算 KV Cache 的块数？

**问题：** 如何动态计算 KV Cache 的块数？

**详细答案：**  
动态块数的本质是：在模型权重与其它运行时开销已占用一部分显存后，用 **`gpu_memory_utilization × 总显存`** 作为目标上限，减去当前与峰值分配，得到 **可给 KV 池的字节数**，再除以单块 `block_bytes` 得下取整整数块数。  
这样避免手工调块数，且与 **warmup 后峰值显存** 对齐，减少 OOM。

**结合 nano-vllm 源码：**  
见 `allocate_kv_cache`：`mem_get_info` 得 `total/free`，`peak` 来自 `memory_stats`，`block_bytes` 由层数、块长、头数、dtype 决定，最后整除得 `num_kvcache_blocks`（`model_runner.py`）。

**面试加分技巧：**  
强调 **warmup 的目的**（峰值统计）与 **`gpu_memory_utilization` 是用户可调的安全余量**。

### 7. KV Cache 的 Block 管理策略有哪些？

**问题：** KV Cache 的 Block 管理策略有哪些？

**详细答案：**  
常见策略包括：**空闲链表/栈** 管理可分配物理块、**引用计数** 支持多序列共享块、**LRU**（部分系统用于驱逐）、以及 **前缀哈希表** 做块级复用。nano-vllm 实现中：**空闲队列 `free_block_ids`、已用集合 `used_block_ids`、哈希表 `hash_to_block_id`**，配合 `ref_count` 在 deallocate 时回收。  
append 时若块满则申请新块；未满则更新最后一块的哈希。

**结合 nano-vllm 源码：**  
`BlockManager.allocate` / `deallocate` / `may_append`（`block_manager.py`）；`Sequence` 提供 `num_blocks`、`last_block_num_tokens` 等（`sequence.py`）。

**面试加分技巧：**  
对比「纯分配释放」与「带共享的引用计数」，说明 **抢占时必须 deallocate 释放物理块**（见调度器一节）。

### 8. 如何将 KV 写入 Cache？nano-vllm 用了什么技术？

**问题：** 如何将 KV 写入 Cache？nano-vllm 用了什么技术？

**详细答案：**  
写入需把本步算出的 `(N, num_heads, head_dim)` 的 K/V 按 **slot_mapping** scatter 到大块缓存的对应位置；若用朴素索引会多次小写，nano-vllm 使用 **Triton JIT kernel** 按 token 并行，每个 program 处理一个 token 的全部 head_dim×heads 展平维度。  
`slot == -1` 时直接返回，兼容占位。

**结合 nano-vllm 源码：**  
`store_kvcache_kernel` 与 `store_kvcache`（`layers/attention.py`）。

**面试加分技巧：**  
提到 **与 FlashAttention 解耦**：先显式写入 cache，再调用 `flash_attn_*` 从 cache 读，职责清晰。

## 二、PagedAttention（6 题）

### 1. 什么是 PagedAttention？与传统 Attention 有什么区别？

**问题：** 什么是 PagedAttention？与传统 Attention 有什么区别？

**详细答案：**  
传统实现容易为每序列预留**连续**长度为 `max_len` 的 KV，显存浪费且难以碎片整理。PagedAttention 将 KV 划为**固定大小物理块**，通过 **Block Table** 把逻辑位置映射到非连续物理块，类似 OS 分页。  
注意力内核需支持 **按 block_table 间接寻址**（FlashAttention-2 的 varlen/with_kvcache 接口）。nano-vllm 在 decode 路径传入 `block_table=context.block_tables`。

**结合 nano-vllm 源码：**  
`prepare_decode` 构造 `block_tables`；`flash_attn_with_kvcache(..., block_table=...)`（`attention.py`）。

**面试加分技巧：**  
用一句话：**逻辑连续、物理离散**，再补 vLLM 论文动机即可。

---

### 2. Block Table 是什么？如何维护逻辑块到物理块的映射？

**问题：** Block Table 是什么？如何维护逻辑块到物理块的映射？

**详细答案：**  
Block Table 是**每个序列**维护的数组：第 i 个逻辑块对应哪个 **物理 block_id**。推理时根据当前 token 在块内偏移算出 **slot**，供写入与注意力寻址。  
维护发生在调度器分配显存时：`allocate` 填满 `seq.block_table`；每步 `may_append` 在块边界扩展新物理块。

**结合 nano-vllm 源码：**  
`Sequence.block_table`；`prepare_block_tables` 填充为张量并 pad -1（`model_runner.py`）。

**面试加分技巧：**  
手画 **逻辑 token 下标 → 块索引 + 块内偏移 → 物理 slot**。

### 3. nano-vllm 的 Block 大小为什么设为 256？

**问题：** nano-vllm 的 Block 大小为什么设为 256？

**详细答案：**  
块大小是 **粒度与开销的折中**：太小则 block table 长、管理开销大；太大则单序列短请求内部碎片多。256 与 FlashAttention / GPU warp 友好形状常见配置一致，且 `Config` 要求 `kvcache_block_size % 256 == 0` 便于对齐内核。  
教学项目常直接沿用 vLLM 默认经验值。

**结合 nano-vllm 源码：**  
`Sequence.block_size = 256`；`Config.kvcache_block_size: int = 256` 与断言（`config.py`、`sequence.py`）。

**面试加分技巧：**  
答「经验默认 + 对齐 + 碎片权衡」，避免断言「唯一最优」。

---

### 4. xxhash 前缀缓存如何实现？链式哈希的设计思路？

**问题：** xxhash 前缀缓存如何实现？链式哈希的设计思路？

**详细答案：**  
单块内对 token 序列做 `xxhash` 得指纹；对**多块前缀**，将**前一块的哈希值作为 seed/prefix** 传入下一块计算，使「同前缀链」自然得到可比较的链式指纹，无需每次对整段前缀重哈希。  
命中条件：`hash_to_block_id[h]` 存在且 **存储的 token 列表与当前块完全一致**（防哈希碰撞）。

**结合 nano-vllm 源码：**  
`compute_hash(token_ids, prefix)`；`allocate` 循环内 `h = self.compute_hash(token_ids, h) if len(token_ids)==block_size else -1`（`block_manager.py`）。

**面试加分技巧：**  
强调 **链式哈希降低长前缀重复计算**，并提 **碰撞用 token 比对兜底**。

### 5. Block 的引用计数（ref_count）有什么作用？

**问题：** Block 的引用计数（ref_count）有什么作用？

**详细答案：**  
当多个序列 **共享**同一物理块（前缀缓存命中）时，不能因一条序列结束就回收块。`ref_count` 记录被多少逻辑使用者持有；降为 0 才归还 `free_block_ids`。  
分配时 `reset` 将 ref 置 1；共享时递增。

**结合 nano-vllm 源码：**  
`allocate` 命中已用块时 `block.ref_count += 1`；`deallocate` 逐块 `--`，为 0 则 `_deallocate_block`（`block_manager.py`）。

**面试加分技巧：**  
类比 **共享指针 / inode 引用**，面试官易共鸣。

---

### 6. allocate 和 deallocate 的完整流程？

**问题：** allocate 和 deallocate 的完整流程？

**详细答案：**  
**allocate**：按序列逻辑块数循环，每块取 token，尝试哈希命中；未命中或碰撞则从 `free_block_ids` 取物理块，`_allocate_block` 初始化，更新哈希表与 `block_table`。  
**deallocate**：逆序遍历 `block_table`，递减引用，为 0 则从 `used` 移除并加入空闲队列，清空序列侧缓存计数与表。

**结合 nano-vllm 源码：**  
`BlockManager.allocate` / `deallocate`（`block_manager.py`）。

**面试加分技巧：**  
说明 **逆序与顺序** 在本实现中 deallocate 用 `reversed` 是为与其它逻辑一致，重点在 **引用计数与空闲队列一致性**。

## 三、连续批处理（5 题）

### 1. 什么是连续批处理？与静态批处理的区别？

**问题：** 什么是连续批处理？与静态批处理的区别？

**详细答案：**  
连续批处理（continuous batching / iteration-level scheduling）指 **每个解码步** 动态选择一批请求执行，请求可随时进出，不必固定同一 batch 大小跑完全程。静态批处理往往要求 batch 内序列等长或 padding 到同一长度，GPU 算力浪费在 padding 上。  
nano-vllm 的 `Scheduler.schedule` 每轮根据资源约束重新组批。

**结合 nano-vllm 源码：**  
`schedule` 返回 `scheduled_seqs, is_prefill`（`scheduler.py`）。

**面试加分技巧：**  
用「**迭代级**而非请求级」概括，并联系 **KV 块动态分配**。

---

### 2. nano-vllm 的调度策略是什么？（Prefill 优先）

**问题：** nano-vllm 的调度策略是什么？

**详细答案：**  
实现为 **先尽可能从 waiting 队列取请求做 prefill**，直到达到 `max_num_seqs` 或 `max_num_batched_tokens` 或块分配失败；仅当本轮 prefill 为空时，才进入 **decode** 分支从 running 队列取序列做单步解码。  
这样优先消化新请求的前向（提高 prompt 吞吐），避免 decode 饿死新请求（视业务可调，也有系统采用混合策略）。

**结合 nano-vllm 源码：**  
`schedule` 先 `while self.waiting ...` 返回 `True`；否则 decode 分支（`scheduler.py`）。

**面试加分技巧：**  
可讨论 **公平性 vs 延迟**：prefill 优先可能让长 decode 队列等待，可提「工业界会加权重/配额」。

---

### 3. 抢占（preempt）机制是什么？何时触发？

**问题：** 抢占（preempt）机制是什么？何时触发？

**详细答案：**  
当 decode 阶段需要为新 token **append 块**但空闲物理块不足时，不能继续执行该步；调度器选择 **抢占**某运行中序列：将其状态置回等待并 **释放 KV 块**，以腾出资源。被抢占序列后续需重新 prefill（实现上已简化在 nano-vllm 语义中）。  
触发条件：`can_append` 为假且无法从其它运行序列让出块时，对某序列 `preempt`。

**结合 nano-vllm 源码：**  
`decode` 分支中 `while not self.block_manager.can_append(seq)` 调用 `preempt`（`scheduler.py`）。

**面试加分技巧：**  
说明 **GPU 显存是硬约束**，抢占是 **背压**手段，与操作系统内存页换出类比。

---

### 4. max_num_seqs 和 max_num_batched_tokens 的作用？

**问题：** max_num_seqs 和 max_num_batched_tokens 的作用？

**详细答案：**  
`max_num_seqs` 限制 **同一 step 并发序列数上限**，控制 decode batch 宽度与调度状态规模。`max_num_batched_tokens` 限制 **prefill 步处理的 token 总数上限**，避免单次 prefill 过长序列占满算力或显存。二者共同约束 scheduler 第一轮 while 循环。  
配置要求 `max_num_batched_tokens >= max_model_len` 以便单序列可跑满长上下文。

**结合 nano-vllm 源码：**  
`schedule` 中条件；`Config` 默认值与断言（`config.py`）。

**面试加分技巧：**  
联系 **OOM 与延迟**：增大并发可提高吞吐但恶化单请求延迟。

---

### 5. 连续批处理如何提升 GPU 利用率？

**问题：** 连续批处理如何提升 GPU 利用率？

**详细答案：**  
通过减少 padding、让每步 **有效矩阵乘尺寸** 更接近硬件饱和，并允许新请求插入与完成请求退出，避免「等最长序列」的空转。与 **PagedAttention** 结合后，显存不随 max_len 线性爆炸，才能支撑大批宽。  
nano-vllm 在 `run_model` 中对较大 decode batch 使用 CUDA Graph 降低 launch 开销，进一步提高有效吞吐。

**结合 nano-vllm 源码：**  
`warmup_model` 用满配置构造序列；`capture_cudagraph`（`model_runner.py`）。

**面试加分技巧：**  
答「减少空算 + 提高 batch 有效 token + 降低 kernel launch」，三层递进。

## 四、FlashAttention（6 题）

### 1. FlashAttention 的核心思想是什么？

**问题：** FlashAttention 的核心思想是什么？

**详细答案：**  
标准注意力在 HBM 读写上瓶颈明显。FlashAttention 通过 **tiling** 在 SRAM 上分块累加 softmax 统计量，减少 HBM 往返，并 **融合** 多个操作。FlashAttention-2 进一步优化并行与 work partitioning。  
nano-vllm 不手写 CUDA attention，而是调用 **flash-attn** 官方算子，与 KV cache 布局对齐。

**结合 nano-vllm 源码：**  
`flash_attn_varlen_func` / `flash_attn_with_kvcache`（`attention.py`）。

**面试加分技巧：**  
一句话：**算得快不如搬得少**（IO 复杂度见后文）。

---

### 2. flash_attn_varlen_func 和 flash_attn_with_kvcache 的区别？

**问题：** flash_attn_varlen_func 和 flash_attn_with_kvcache 的区别？

**详细答案：**  
`varlen_func` 面向 **一批不等长序列拼接成的长张量**，用 `cu_seqlens` 描述边界，适合 **prefill** 多序列并行且无显式 KV cache 张量（或 K/V 为本次计算出的张量）。  
`with_kvcache` 面向 **decode**：Q 为新 token，K/V 来自 **持久化 cache**，并配合 `cache_seqlens`、`block_table` 做分页寻址。  
nano-vllm 严格按 `is_prefill` 分支选择。

**结合 nano-vllm 源码：**  
`Attention.forward` 分支（`attention.py`）。

**面试加分技巧：**  
强调 **数据布局与 API 契约不同**，不能混用。

---

### 3. cu_seqlens_q 和 cu_seqlens_k 的含义？

**问题：** cu_seqlens_q 和 cu_seqlens_k 的含义？

**详细答案：**  
二者均为 **前缀和数组**（长度 N+1），`cu_seqlens[i+1]-cu_seqlens[i]` 为第 i 个序列在拼接维上的长度。Q 与 K 可以不同：例如 **前缀缓存** 命中时，本轮只算新 token 的 Q（短），但 K 侧包含完整上下文（长），故 `cu_seqlens_k[-1] > cu_seqlens_q[-1]`。  
nano-vllm 在 `prepare_prefill` 中分别累积 `seqlen_q` 与 `seqlen_k`。

**结合 nano-vllm 源码：**  
`prepare_prefill` 循环内更新；prefix 时准备 `block_tables`（`model_runner.py`）。

**面试加分技巧：**

### 4. 为什么 Prefill 用 varlen_func 而 Decode 用 with_kvcache？

**问题：** 为什么 Prefill 用 varlen_func 而 Decode 用 with_kvcache？

**详细答案：**  
Prefill 主要算 **大量新 token** 的注意力，适合 varlen 批处理；且此时 K/V 往往从计算直接流入或需灵活布局。Decode 每步 **每序列仅 1 个 query**，重复从 HBM 读整段 K/V 已优化为 **cache + 单步接口**，`with_kvcache` 把 **写入与读取** 与分页信息封装好。  
nano-vllm 还在 prefill 写入阶段用 Triton 显式 `store_kvcache`，再读 cache 做注意力，路径清晰。

**结合 nano-vllm 源码：**  
`store_kvcache` + 两分支（`attention.py`）。

**面试加分技巧：**  
从 **计算强度与内存带宽角色互换** 角度解释 prefill vs decode 瓶颈差异。

### 5. FlashAttention v1 / v2 / v3 的演进？

**问题：** FlashAttention v1 / v2 / v3 的演进？

**详细答案：**  
v1 提出 IO-aware tiling；v2 改进并行策略、减少非 matmul 工作，支持更长序列更高效；v3 在 Hopper 等架构上利用 **异步 TMA、warp-specialized** 等进一步压榨硬件（细节依版本与论文更新）。

### 6. FlashAttention 的 IO 复杂度分析？

**问题：** FlashAttention 的 IO 复杂度分析？

**详细答案：**

## 五、张量并行（6 题）

### 1. 什么是张量并行？与数据并行、流水线并行的区别？

**问题：** 什么是张量并行？与数据并行、流水线并行的区别？

**详细答案：**  
**张量并行（TP）** 将单层内大矩阵乘按列/行切到多 GPU，各卡存部分权重，前向通过通信拼完整结果。**数据并行（DP）** 每卡完整模型、不同 batch，梯度同步。**流水线并行（PP）** 按层切分到不同设备，微批次流水。  
nano-vllm 实现 **单机多卡 TP**（`dist.init_process_group`），无 PP。

**结合 nano-vllm 源码：**  
`ModelRunner` 初始化 NCCL；`linear.py` 中 `dist.get_world_size()`。

**面试加分技巧：**  
一句：**TP 切层内张量，PP 切层间，DP 切 batch**。

---

### 2. 列并行（ColumnParallel）和行并行（RowParallel）的原理？

**问题：** 列并行和行并行的原理？

**详细答案：**  
线性层 `Y = XA^T`：列并行将 **输出维** 切分，每卡算一部分输出，若需完整输出要 concat 或后续层接受分片输入；行并行将 **输入维** 切分，每卡算部分内积，**结果需 all-reduce 求和** 得到完整输出。  
Transformer 中常见模式：**Attention QKV 列并行**，**输出投影行并行**，减少通信次数。

**结合 nano-vllm 源码：**  
`ColumnParallelLinear` 切 `output_size`；`RowParallelLinear` 切 `input_size` 且 forward 内 `all_reduce`（`linear.py`）。

**面试加分技巧：**  
手写 **小矩阵分块乘法** 示意。

---

### 3. QKVParallelLinear 的分片策略？

**问题：** QKVParallelLinear 的分片策略？

**详细答案：**  
将 Q、K、V 三部分 **拼接成一个大 output 维**，再按列并行规则切到各 TP rank；加载权重时按 **shard_id**（q/k/v）把对应段写入本卡参数子块。  
头数与 KV 头数需整除 TP。

**结合 nano-vllm 源码：**  
`QKVParallelLinear` 的 `output_size` 与 `weight_loader`（`linear.py`）。

**面试加分技巧：**  
强调 **合并线性层减少 kernel 次数** 的工程收益。

---

### 4. AllReduce 通信的开销分析？

**问题：** AllReduce 通信的开销分析？

**详细答案：**  
RowParallel 后每卡得到部分和，需 all-reduce 得到完整向量。复杂度随 **向量长度、卡数、网络拓扑** 变化；NCCL 会选 ring/tree 等算法。TP 对 **小 batch decode** 时通信占比可能上升，是混合并行策略要权衡的点。  
nano-vllm 在 `RowParallelLinear` 与 `VocabParallelEmbedding` 使用 all_reduce。

**结合 nano-vllm 源码：**  
`dist.all_reduce(y)`（`linear.py`）；Embedding 前向（`embed_head.py`）。

**面试加分技巧：**  
提到 **与 batch size 和 hidden size 成比例**，并联系 **多卡扩展性非线性**。

### 5. VocabParallelEmbedding 如何实现词表并行？

**问题：** VocabParallelEmbedding 如何实现词表并行？

**详细答案：**  
将词表维切成多段，每卡只存 **一段 embedding**。前向时：若 token id 落在本卡段内则查表得到向量，否则为 0；再 **all_reduce** 聚合各卡贡献，得到完整 embedding（掩码避免错误累加）。  
nano-vllm 用 `mask` 与 `vocab_start_idx`/`vocab_end_idx` 判断。

**结合 nano-vllm 源码：**  
`VocabParallelEmbedding.forward`（`embed_head.py`）。

**面试加分技巧：**  
对比 **查表是稀疏的**：每 token 只命中一个分区，故用 mask+reduce。

### 6. ParallelLMHead 如何用 gather 收集 logits？

**问题：** ParallelLMHead 如何用 gather 收集 logits？

**详细答案：**  
LM Head 本质是 `hidden @ W^T`，列并行下每卡只持有部分词表列，得到 **部分 logits**。需在 **rank0** 上 `gather` 各卡 logits 再 `cat` 成完整词表维度。Prefill 时还需按 `cu_seqlens_q` 取 **每序列最后一个 query** 位置，避免对中间位置采样。  
nano-vllm 使用 `dist.gather` 而非 all_gather 变体，由 rank0 收集。

**结合 nano-vllm 源码：**  

```56:66:nano-vllm-main/nanovllm/layers/embed_head.py
    def forward(self, x: torch.Tensor):
        context = get_context()
        if context.is_prefill:
            last_indices = context.cu_seqlens_q[1:] - 1
            x = x[last_indices].contiguous()
        logits = F.linear(x, self.weight)
        if self.tp_size > 1:
            all_logits = [torch.empty_like(logits) for _ in range(self.tp_size)] if self.tp_rank == 0 else None
            dist.gather(logits, all_logits, 0)
            logits = torch.cat(all_logits, -1) if self.tp_rank == 0 else None
```

**面试加分技巧：**  
说明 **prefill 取 last token** 与 **decode 每序列单 token** 的差异。

Prefill 的 **序列长度与拼接长度** 变化大，且常触发 `input_ids.size(0) > 512` 等分支；decode 步 **batch 内 token 数等于序列数**，形状相对稳定。nano-vllm 在 `run_model` 中：`is_prefill` 或 `enforce_eager` 或 batch token 数>512 时走 eager。

### 3. graph_bs 的离散 batch size 设计有什么考虑？

**问题：** graph_bs 的离散 batch size 设计有什么考虑？

**详细答案：**  
实际并发序列数落在任意整数，若每个 bs 都录一张图，**编译与存储开销**过大。离散化用 **不小于当前 bs 的最小预录 bs**（`next(x for x in graph_bs if x >= bs)`），在 `graph_vars` 中只填充前 bs 个槽位，其余不参与有效计算。

**结合 nano-vllm 源码：**

**面试加分技巧：**

### 4. CUDA Graph 的限制是什么？

**问题：** CUDA Graph 的限制是什么？

## 七、RoPE（4 题）

### 1. RoPE 的数学原理？

**问题：** RoPE 的数学原理？

**详细答案：**  
旋转位置编码将 query/key 向量两两维度成对，视作复平面上的向量，乘以位置相关的旋转因子，使注意力内积仅依赖 **相对位置**。实现上常用 **cos/sin** 缓存与 `apply_rotary_emb`。  
性质：对长序列外推存在局限，故有 NTK、YaRN 等改进。

**结合 nano-vllm 源码：**  
`RotaryEmbedding` 与 `apply_rotary_emb`（`rotary_embedding.py`）。

**面试加分技巧：**  
一句话：**相对位置编码进内积**，再补「旋转矩阵」直觉。

---

### 2. inv_freq 是如何计算的？

**问题：** inv_freq 是如何计算的？

**详细答案：**  
对维度索引按偶数步采样，令 `inv_freq = 1 / (base ** (i/rotary_dim))`，不同维度对应不同频率，形成多频正弦余弦基；位置 t 与 inv_freq 外积得 `freqs`，再 cos/sin。  
`base` 通常为超参（如 rope_theta）。

**结合 nano-vllm 源码：**  

```29:34:nano-vllm-main/nanovllm/layers/rotary_embedding.py
        inv_freq = 1.0 / (base**(torch.arange(0, rotary_dim, 2, dtype=torch.float) / rotary_dim))
        t = torch.arange(max_position_embeddings, dtype=torch.float)
        freqs = torch.einsum("i,j -> ij", t, inv_freq)
        cos = freqs.cos()
        sin = freqs.sin()
```

**面试加分技巧：**  
指出 **与 Transformer 正弦位置编码的频率形式类比**。

---

### 3. apply_rotary_emb 的实现原理？

**问题：** apply_rotary_emb 的实现原理？

**详细答案：**  
将最后一维拆成两半 `(x1, x2)`，按旋转公式 `y1 = x1*cos - x2*sin`，`y2 = x2*cos + x1*sin`，再拼接；与缓存的 cos/sin（来自 positions 索引）相乘。  
使用 float 中间精度再 cast 回原 dtype 减少数值问题。

**结合 nano-vllm 源码：**  
`apply_rotary_emb`（`rotary_embedding.py`）。

**面试加分技巧：**  
提到 **只对 q、k 应用，不对 v**（标准做法）。

### 4. RoPE 的长度外推方法有哪些？（NTK-aware、YaRN 等）

**问题：** RoPE 的长度外推方法有哪些？

**详细答案：**  
常见包括：**NTK-aware**（缩放 base 或频率以减缓高频衰减）、**YaRN**（分段混合缩放与注意力温度修正）、**位置插值 PI** 等。nano-vllm 的 `get_rope` 中 **`rope_scaling` 直接 assert None**，教学代码未实现这些变体，但模型配置可传入占位。

**MHA**：Q/K/V 头数相同。**MQA**：多 Q 共享单一 K/V。**GQA**：K/V 头数为 Q 头数的约数，组内共享。

**问题：** nano-vllm 中 GQA 如何实现？

**详细答案：**

### 3. GQA 的 KV 头数与 Q 头数的比例关系？

**问题：** GQA 的 KV 头数与 Q 头数的比例关系？

**详细答案：**

## 九、Sampler / Decoding 策略（5 题）

### 1. 温度采样的原理？

**问题：** 温度采样的原理？

**详细答案：**  
将 logits 除以温度 T：T>1 分布更平（探索多）；T<1 更尖（更确定）。再对分布采样。数学上等价于调整 softmax 的熵。  
nano-vllm 在 sampler 中对 logits 按温度缩放后 softmax。

**结合 nano-vllm 源码：**  
`logits.float().div_(temperatures.unsqueeze(dim=1))` 后 `softmax`（`sampler.py`）。

**面试加分技巧：**  
提 **数值稳定性**：先减 max 再 softmax（若手写）；本项目依赖 PyTorch softmax。

---

### 2. nano-vllm 的 Gumbel-max 采样技巧？

**问题：** nano-vllm 的 Gumbel-max 采样技巧？

**详细答案：**  
Gumbel-max 技巧：对类别 k 独立采样 Gumbel 噪声 `g_k`，取 `argmax(log p_k + g_k)` 等价于从 softmax(p) 采样。实现上等价变体：**指数分布**噪声与概率相除再 argmax（与 Gumbel 存在可转换关系）。  
nano-vllm 使用：`probs.div_(exponential(1)).argmax`。

**结合 nano-vllm 源码：**

### 3. Top-k / Top-p 采样原理？

**问题：** Top-k / Top-p 采样原理？

**详细答案：**  
**Top-k**：只在概率最高的 k 个 token 上重新归一化并采样。**Top-p（nucleus）**：按概率从大到小累加至超过阈值 p，截取前缀再归一化采样。  
nano-vllm **未实现** top-k/top-p，仅基础温度+Gumbel-max式采样，面试可说明扩展点。

**结合 nano-vllm 源码：**  
`sampler.py` 无 top-k/p。

**面试加分技巧：**  
诚实 + **若我做会加在 softmax 后作为可插拔策略**。

---

### 4. 什么是投机解码（Speculative Decoding）？

**问题：** 什么是投机解码（Speculative Decoding）？

**详细答案：**  
用小模型或草稿模型 **多步预测**候选 token，大模型并行验证，接受则加速生成，拒绝则回退。可降延迟但增加系统复杂度。nano-vllm **未实现**。

**面试加分技巧：**  
提 **验证并行化** 与 **接受率** 是关键。

---

### 5. 为什么 nano-vllm 不支持贪心解码？

**问题：** 为什么 nano-vllm 不支持贪心解码？

**详细答案：**  
贪心等价 **温度→0** 或 argmax 直接取最大；项目通过 `SamplingParams` **禁止极低温度**，断言 `temperature > 1e-10`，从源头避免「声称支持贪心」却与采样实现不一致；且 sampler 路径是 **随机采样**实现。若要贪心应改代码走 `argmax` 分支。

**结合 nano-vllm 源码：**  

```10:12:nano-vllm-main/nanovllm/sampling_params.py

        assert self.temperature > 1e-10, "greedy sampling is not permitted"
```

**面试加分技巧：**  
表述为 **教学项目简化与 API 一致性**，避免面试官追问数值 corner case。

## 十、推理优化（5 题）

### 1. torch.compile 的优化原理？nano-vllm 对哪些模块使用了它？

**问题：** torch.compile 的优化原理？nano-vllm 对哪些模块使用了它？

**详细答案：**  
`torch.compile` 通过 Dynamo 捕获 FX Graph，再交给后端（如 Inductor）做 **算子融合、内存规划、内核选择** 等。对 **小算子密集**模块收益明显。  
nano-vllm 用于 `Sampler.forward` 与 `RotaryEmbedding.forward`（装饰器 `@torch.compile`）。

**结合 nano-vllm 源码：**  
`sampler.py`、`rotary_embedding.py`。

**面试加分技巧：**  
提 **首次编译延迟** 与 **shape 变动** 的权衡。

### 2. 量化（INT8 / INT4）的基本原理？

**问题：** 量化（INT8 / INT4）的基本原理？

**详细答案：**  
将浮点权重/激活映射到离散整数，配合 **scale/zero_point**；推理用整数矩阵乘或混合精度。INT4 更省显存但对精度与内核要求更高。  
nano-vllm 示例为 bf16/fp16 路径，**未内置 GPTQ/AWQ 加载**，可对比 vLLM。

**面试加分技巧：**  
点出 **PTQ vs QAT**。

---

### 3. AWQ 和 GPTQ 的区别？

**问题：** AWQ 和 GPTQ 的区别？

**详细答案：**  
**GPTQ**：逐列量化、基于 Hessian 的误差补偿，偏离线压缩。**AWQ**：强调 **激活感知**、保护重要权重通道，常与平滑因子结合。二者都是 PTQ 主流路线。  
nano-vllm 未实现，答概念即可。

**面试加分技巧：**  
一句话：**一个更 Hessian/Greedy，一个更 activation-aware**。

---

### 4. 算子融合（Operator Fusion）的原理？

**问题：** 算子融合的原理？

**详细答案：**  
将相邻算子合并为单 kernel，减少中间张量读写与 kernel launch。FlashAttention、LayerNorm+Linear 融合均属此类。  
nano-vllm 依赖 flash_attn 与 `torch.compile` 间接融合。

**面试加分技巧：**  
用 **访存与 launch 次数** 两个维度说。

---

### 5. pin_memory 和 non_blocking 传输的作用？

**问题：** pin_memory 和 non_blocking 传输的作用？

**详细答案：**  
`pin_memory=True` 分配页锁定内存，**异步 DMA** 到 GPU 更快；`non_blocking=True` 配合 stream 允许 CPU 继续准备下一批。  
nano-vllm 在 tensor 化 input、positions、cu_seqlens 等时使用二者。

**结合 nano-vllm 源码：**  
如 `torch.tensor(..., pin_memory=True).cuda(non_blocking=True)`（`model_runner.py`）。

**面试加分技巧：**  
提醒 **同步点**：需在用到 GPU 数据前 `synchronize` 或使用默认同步行为。

## 十一、Triton 编程（4 题）

### 1. Triton 和 CUDA 的区别？

**问题：** Triton 和 CUDA 的区别？

**详细答案：**  
CUDA 用 C++/PTX 细粒度控制线程与内存；Triton 是 Python DSL，**以 tile 为单位**描述计算，编译器负责生成高效内核，上手更快。nano-vllm 用 Triton 写 KV store，无需手写大量索引算术的 CUDA。

**结合 nano-vllm 源码：**  
`import triton`（`attention.py`）。

**面试加分技巧：**  
说清 **适用边界**：复杂不规则控制流可能仍用 CUDA。

---

### 2. @triton.jit 的工作原理？

**问题：** @triton.jit 的工作原理？

**详细答案：**  
装饰器将 Python 函数 **JIT 编译**为 GPU 内核；首次调用根据 **模板参数（constexpr）** 与网格配置编译缓存。函数内用 `tl` 命名空间表达 SIMD 操作。

**结合 nano-vllm 源码：**  
`@triton.jit` 修饰 `store_kvcache_kernel`。

**面试加分技巧：**  
提 **编译缓存与首次调用慢**。

---

### 3. tl.program_id、tl.arange 的含义？

**问题：** tl.program_id、tl.arange 的含义？

**详细答案：**  
`tl.program_id(axis)` 返回当前 **program（类似 block）** 在指定轴上的索引；`tl.arange` 生成 0..N-1 的一维向量，用于构造 load/store 指针偏移。  
在 `store_kvcache_kernel` 中 `idx = tl.program_id(0)` 对应第几个 token。

**结合 nano-vllm 源码：**  
`store_kvcache_kernel` 内（`attention.py`）。

**面试加分技巧：**  
类比 CUDA 的 `blockIdx` 与 `threadIdx` 分工（Triton 层次不同）。

---

### 4. store_kvcache_kernel 的设计思路？

**问题：** store_kvcache_kernel 的设计思路？

**详细答案：**  
每个 program 负责一个 token：读 `slot_mapping` 得全局 slot，若无效则返回；将 K/V 指针按 stride 展平为长度 D=`num_heads*head_dim` 的向量 load，再写入 cache 的 `slot*D` 偏移。  
**合并头维**为一条向量，简化寻址；`D` 为 `constexpr` 便于编译器优化。

**结合 nano-vllm 源码：**  
完整 kernel（`attention.py`）。

**面试加分技巧：**  
强调 **与 FlashAttention 解耦的显式写入**，便于调试 KV 布局。

### 2. SwiGLU 激活函数的原理？（SiLU 门控）

**问题：** SwiGLU 激活函数的原理？

**详细答案：**  
SwiGLU 将隐藏层分为 gate 与 up 两路：`SiLU(gate) * up`，再经 down 投影。`SiLU` 即 sigmoid 加权 x。比单一 ReLU/GELU 表达力更强，参数略增。  
nano-vllm 用 `MergedColumnParallelLinear` 输出两倍中间维，再 `SiluAndMul`。

**结合 nano-vllm 源码：**  
`Qwen3MLP` + `SiluAndMul`（`qwen3.py`、`activation.py`）。

**面试加分技巧：**  
画 **门控乘法门**。

---

### 3. RMSNorm vs LayerNorm？

**问题：** RMSNorm vs LayerNorm？

**详细答案：**  
RMSNorm 不减均值，仅按 **均方根**缩放，参数更少、计算更省；在 LLM 中常与 Pre-Norm 搭配稳定训练。LayerNorm 减均值加方差归一化。  
nano-vllm 全用 `RMSNorm`。

**结合 nano-vllm 源码：**  
`layernorm.py`（如存在）；`Qwen3DecoderLayer` 引用。

**面试加分技巧：**  
提 **Pre-Norm 缓解梯度** 与推理无关但常追问。

---

### 4. Pre-Norm vs Post-Norm？

**问题：** Pre-Norm vs Post-Norm？

**详细答案：**  
Pre-Norm 将 LN 放在子层 **输入**，训练更稳定、易堆深；Post-Norm 将 LN 放在 **输出**，经典 Transformer 原始设计。现代 LLM 多 Pre-Norm。  
nano-vllm `Qwen3DecoderLayer`：`input_layernorm` 在注意力前，`post_attention_layernorm` 在 MLP 前，属 Pre-Norm 结构。

**结合 nano-vllm 源码：**  
`forward` 参数 `residual`（`qwen3.py`）。

**面试加分技巧：**  
结合 **残差分支** 说明。

### 2. SharedMemory 多进程通信的实现？

**问题：** SharedMemory 多进程通信的实现？

**详细答案：**  
rank0 `write_shm`：将 pickle 后的 `[method_name, args]` 写入共享内存前 4 字节长度前缀；对子进程 `event.set()`。子进程 `read_shm`：`event.wait()` 后读长度与 payload，`pickle.loads`，`clear` 事件。

**结合 nano-vllm 源码：**

### 3. Event 同步机制的作用？

**问题：** Event 同步机制的作用？

**详细答案：**  
保证 **写端完成后再读**，避免读到半包数据。多子进程时对 `event` 列表统一 `set`，实现 **一次调用唤醒所有 worker**。

### 4. 如何计算推理吞吐量？

**问题：** 如何计算推理吞吐量？

**详细答案：**  
**Prefill 吞吐**：处理的总 prompt token 数 / 耗时（tokens/s）。**Decode 吞吐**：生成 token 数 / 耗时；多序列时按 **总生成 token** 计。首 token 延迟（TTFT）常单独统计。
