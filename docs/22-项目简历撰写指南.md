## 一、简历项目经验模板（多个版本）

### 版本一：校招简历（简洁版）

适合：一页简历、项目较多、每条不宜过长。

```
项目名称：基于 nano-vllm 的大模型推理引擎深度学习与优化
项目时间：2026.xx - 2026.xx
技术栈：Python, PyTorch, CUDA, Triton, FlashAttention, NCCL
项目描述：深入学习并优化了 nano-vllm 轻量级大模型推理引擎（约 1200 行 Python，社区高 star 教学向实现），掌握了生产级推理框架的核心技术。
核心工作：
- 深入理解并分析了 PagedAttention 内存管理机制，掌握了基于 xxhash 的前缀缓存实现
- 学习了连续批处理调度策略，理解 Prefill 优先与 Decode 阶段对 running 队列的调度方式
- 分析了张量并行（TP）实现，包括列/行并行 Linear 和 QKVParallelLinear
- 理解了 CUDA Graph 优化的录制-重放机制，掌握了 Triton Kernel 编写
- 通过 bench.py 复现性能测试，在相同硬件与模型下与 vLLM 量级对齐（以本人实测为准）
```

**写法提示**：把「学习了」改成「复现/实现/调试了」更主动；若你以**读源码为主**，可写「源码级梳理并实现局部实验」，避免虚构「从零开发全引擎」。

### 版本二：社招简历（详细版）

适合：工作年限 1–3 年、推理/Infra 岗、需要体现**模块边界与工程决策**。

> **项目名称**：轻量级 LLM 推理引擎（nano-vllm 架构实践与性能分析）  
> **项目时间**：2026.xx – 2026.xx  
> **技术栈**：Python、PyTorch、CUDA、FlashAttention-2、Triton、NCCL、torch.compile、SharedMemory  
> **项目描述**（示例 bullet，按需删减）：
> - 基于 nano-vllm（~1200 行）梳理并实践 **vLLM 类架构**：调度（`Scheduler`）、分页 KV（`BlockManager`）、执行（`ModelRunner`）、注意力与 KV 写入（`Attention` + Triton）、多卡 TP（`linear.py` / NCCL）。
> - **显存与分页**：实现/调试 `BlockManager` 的块分配、`allocate`/`may_append`/`deallocate`、xxhash 链式哈希与前缀命中；理解 `block_table` 与物理块池 `free_block_ids` 的映射关系。
> - **连续批处理**：实现或验证 `schedule()` 中 **Prefill 与 Decode 分两阶段**、显存不足时 `preempt` 与 `can_append` 的交互；能说明与工业界 FCFS/LIFO 抢占表述的对应关系。
> - **计算与图优化**：Prefill 走 `flash_attn_varlen_func`，Decode 走 `flash_attn_with_kvcache`；Decode 路径配合 `capture_cudagraph` 多 batch size 录制与 `graph.replay()`；热点处配合 `torch.compile`（如采样、激活等，以代码为准）。
> - **结果**：输出可复现的 benchmark 报告（模型、序列长度、batch、硬件），能对比 eager / CUDA Graph、与 vLLM 同配置差异；沉淀问题定位笔记（如 slot_mapping、prefix cache 分支）。

**社招加分句**：单独一行写**你独有的贡献**：例如「增加显存 profiling 脚本」「修复某边界 case」「扩展某配置项」——避免与纯开源使用者同质化。

## 二、简历撰写原则

### 2.1 STAR 法则应用

- **S（情境）**：一句话——大模型推理要同时解决显存、吞吐、延迟；工业界常用 vLLM 类架构。  
- **T（任务）**：你的目标是「掌握/实现/评测」什么（具体到模块）。  
- **A（行动）**：用动词开头：**阅读/调试/实现/对比/Profiling**，对应到 `block_manager.py`、`scheduler.py` 等真实文件名。  
- **R（结果）**：量化或**可核验**结果：吞吐、TTFT、代码行数、文档页数、issue 链接；没有数字就写「能白板讲清数据流」。

简历正文可压缩为「行动 + 结果」，面试再用完整 STAR 展开（见 `23-STAR面试稿.md`）。

### 2.2 量化成果

优先使用：**代码规模、实验环境、指标名称、对比对象**。

- 代码：「通读约 1200 行核心代码」「梳理 4 大模块」比「熟悉代码」强。  
- 性能：「RTX 4070、Qwen3-0.6B、固定 prompt 长度」比单独一个 tok/s 数字强。  
- 对比：「相对 eager 模式」「相对 vLLM 同配置」比「效果很好」强。

避免：**无法自证的数量级**（如「显存一定提升 40%」）——除非你有测量方法并在面试能讲清。

### 2.3 技术关键词优化（ATS 友好）

- 在 bullet **前部**自然嵌入 JD 高频词：`PagedAttention`、`KV Cache`、`Continuous Batching`、`Tensor Parallel`、`FlashAttention`、`CUDA Graph`。  
- 同一段里避免重复堆砌 10 遍；可用「负责 KV Cache 分页（PagedAttention）」一次合并表达。  
- 文件名、论文名、库名拼写正确：`FlashAttention-2`、`NCCL`、`Triton`。

### 2.4 避免的错误

| 避免 | 原因 | 建议 |
|------|------|------|
| 「抄了 vLLM / 只跑 demo」 | 负面暗示 | 写「对标 vLLM 架构的教学实现」「源码级学习」 |
| 堆砌关键词无落地 | ATS 可能过，面试必挂 | 每个词对应一个你能讲 2 分钟的点 |
| 虚构 star 数、排名 | 可核查 | 写「社区流行的教学项目」或准确引用 |
| 全是「了解/熟悉」 | 显得被动 | 改为「调试过」「画过数据流」「写过脚本」 |
| 与 JD 无关的一长串技术 | 噪音 | 按岗位删减，推理岗保留 CUDA/调度/KV |

## 四、简历 QA

### Q1：「你只是学习了这个项目，没有自己写代码怎么办？」

**建议**：学习本身可以写进简历，但要**诚实 + 增量**。  
- 简历写「源码阅读、调试、实验复现」，不写「独立原创全引擎」。  
- 在开源或本地 fork 上完成**小改进**：单测、日志、配置项、性能脚本、注释 PR。  
- 面试用 STAR：你**解决的具体 bug/疑问**（例如 slot_mapping 与 block_table 对应关系）比空泛「认真学了」更有说服力。

### Q2：「面试官会问什么？」

- **基础**：prefill vs decode、KV Cache 为何需要、PagedAttention 解决什么问题。  
- **源码**：`schedule` 两阶段、`can_append` 与 `len(seq) % block_size`、`deallocate` 为何倒序减引用计数。  
- **扩展**：量化、投机解码、PD 分离、Chunked Prefill——见教程 `21-面试八股文大全.md` 与 `24-面试问题全集-STAR回答.md`。

### Q3：「如何证明你真的理解了？」

- **准备 STAR 故事**：每个核心模块 1 个「遇到问题—查代码—验证」的故事。  
- **白板数据流**：从请求到 `Sequence`、到 `block_table`、到 FlashAttention 接口参数。  
- **可演示**：本地跑通 `example.py` / `bench.py`，能解释每一步日志与显存变化。
